# 09 - 追踪 / 交互题详解

对应 [02-PICO面试题库.md](./02-PICO面试题库.md)「追踪 / 交互方向」部分。PICO 交互体验的核心是 **低延迟、高鲁棒性的 6DoF 追踪**，涵盖裸手、手柄、面部等多模态感知。

> **GitHub 阅读：** 公式使用 `$...$` / `$$...$$` 格式。

---

## 总览：XR 交互追踪体系

```
头显位姿（SLAM / VIO）────┐
                         ├──→ 统一世界坐标系 → 渲染 / 交互逻辑
手柄 6DoF（光学+IMU+CV）──┤
裸手 6DoF / 21 关键点 ────┤
面部 / 眼动追踪 ──────────┘
```

**核心指标：**

| 指标 | 典型目标 | 说明 |
|------|----------|------|
| **MTP 延迟** | < 20–30 ms | Motion-to-Photon，动作到显示 |
| **追踪频率** | 60–90 Hz+ | 与显示刷新率匹配或更高 |
| **丢失率** | < 0.1% 帧 | 快速挥手、遮挡场景 |
| **位置精度** | mm 级 | 手柄 < 5 mm，手部关键点视场景 |
| **角度精度** | 0.5–1° | 旋转追踪 |

## 一、裸手追踪（Hand Tracking）

### Q1：21 关键点 pipeline 是怎样的？如何在 <30ms 内完成？

#### 1.1 什么是 21 关键点？

每只手用 21 个 3D 关节点表示（MediaPipe / MANO 标准）：

```
        食指   中指   无名指  小指
          ↑     ↑      ↑      ↑
手腕 ── 掌心骨架 ── 各指三节关节
```

| 索引区域 | 数量 | 说明 |
|----------|------|------|
| 手腕 | 1 | Wrist |
| 拇指 | 4 | CMC, MCP, IP, TIP |
| 其余四指 | 各 4 | MCP, PIP, DIP, TIP |
| **合计** | **21** | 每只手 |

输出：每点 $(x, y, z)$ + 可选置信度；双手即 42 点。

#### 1.2 典型 Pipeline

```
[相机采集] 2–3 ms
    ↓
[ISP / 畸变校正] 1–2 ms
    ↓
[手部 ROI 检测] 3–5 ms        ← 全图 vs 热区跟踪
    ↓
[21 关键点回归 / 分类] 8–15 ms  ← 主网络，INT8 on NPU
    ↓
[3D 抬升 / 深度估计] 2–4 ms    ← 立体 / 单目 + 骨骼约束
    ↓
[时序滤波] 1–2 ms              ← One-Euro / Kalman
    ↓
[手势识别 / 交互逻辑] 1–2 ms
    ↓
[提交渲染] ────────────────── 总计目标 < 25–30 ms
```

#### 1.3 检测 + 估计两阶段 vs 端到端

| 方案 | 流程 | 优点 | 缺点 |
|------|------|------|------|
| **两阶段** | 手检测 bbox → ROI 内关键点 | ROI 小，网络轻 | 检测失败则级联失败 |
| **端到端** | 全图直接出 21 点 | 简单 | 算力大，远距离手小 |
| **热区跟踪** | 首帧检测，后续跟踪上一帧扩展 ROI | 平均最快 | 快速出画需重检测 |

**工程常用：** 检测（低频）+ 跟踪（高频）混合。

#### 1.4 网络结构选型

| 模型 | 特点 |
|------|------|
| **MediaPipe Hands** | 手掌检测 + 关键点，移动端成熟 |
| **轻量 CNN / MobileNet** | 自定义 ROI 回归 |
| **HRNet / HigherHRNet** | 高精度 2D，需 3D lifting |
| **直接 3D 回归** | 相机坐标系输出，避免 lifting 误差 |
| **图卷积 GCN** | 利用手部骨骼拓扑约束 |

**3D 获取方式：**
- **双目立体：** 三角测量，精度好，需标定与同步  
- **单目 + 深度网络：** MiDaS 类或专用 hand depth  
- **骨骼长度约束：** MANO 先验，优化 2D→3D 一致性  

#### 1.5 延迟优化（面试必答）

| 手段 | 收益 |
|------|------|
| INT8 量化 + NPU | 推理 2–4× 加速 |
| ROI 裁剪 | 输入从 640² 降到 128² |
| 隔帧检测 + 中间帧跟踪 | 检测每 3 帧一次 |
| 模型蒸馏 | 小网络保精度 |
| Pipeline 并行 | 采集 N+1 与推理 N 重叠 |
| 固定分辨率输入 | 避免 dynamic resize |

**延迟预算示例（30ms）：**

| 模块 | ms |
|------|-----|
| 采集 + 预处理 | 4 |
| ROI 检测 | 4 |
| 21 点网络 | 14 |
| 3D + 滤波 | 5 |
| 提交 | 3 |
| **合计** | **30** |

### Q2：遮挡和快速运动怎么处理？

#### 2.1 遮挡类型

| 类型 | 表现 | 策略 |
|------|------|------|
| **自遮挡** | 握拳时手指不可见 | 3D 骨骼约束 + MANO 先验补全 |
| **物体遮挡** | 手被桌子挡住 | 时序预测 + 重检测 |
| **双手互遮** | 两手交叠 | 双手联合建模或独立 track ID |
| **出画** | 手移出 FOV | 边界预测 + 丢失状态机 |

#### 2.2 快速运动

**问题：**
- 运动模糊 → 关键点漂移  
- 帧间位移大 → ROI 跟踪丢失  
- Rolling shutter → 几何形变  

**对策：**

```
1. 提高曝光快门速度（与 ISP 协作，可能增加噪声）
2. 扩大 ROI 搜索范围（按速度动态扩展）
3. IMU 辅助（头显 IMU 预测手相对运动趋势，若有外接手环更好）
4. 高帧率相机（90Hz 采集，60Hz 输出插值）
5. 时序模型：利用前 N 帧做 trajectory 预测
6. 运动模糊感知网络 / 去模糊预处理（算力换精度）
```

#### 2.3 时序滤波

**One-Euro Filter：** 自适应低通 — 静止时平滑，快速运动时减少滞后。

**Kalman Filter：** 状态 = 位置 + 速度，预测 + 观测更新。

**学习式平滑：** 小型 RNN/LSTM 对关键点序列去抖（注意引入延迟）。

**权衡：** 滤波越强越平滑，但 **MTP 延迟**和**快速响应**变差；手势识别（捏合、挥手）需要低滞后。

#### 2.4 置信度与状态机

```
每关键点输出 confidence c_i

if mean(c) < θ_lost:
    state = LOST → 触发全图重检测
elif mean(c) < θ_degraded:
    state = PREDICT → 仅用 Kalman 预测，不更新观测
else:
    state = TRACKING → 正常更新
```

**面试一句话：** 遮挡靠骨骼先验 + 时序预测；快运动靠高帧率 + 动态 ROI + 轻滤波。

## 二、手柄 6DoF 追踪

### Q3：光学 + IMU + CV 融合（Centaur 多模态）怎么工作？

#### 3.1 6DoF 是什么？

**6 Degrees of Freedom：** 3D 位置 $(x,y,z)$ + 3D 旋转（pitch, yaw, roll 或四元数 $q$）。

手柄追踪需要实时输出位姿，供 Unity/Unreal 渲染虚拟手柄。

#### 3.2 传统光学追踪（Outside-in / Inside-out）

**PICO 头显内向外（Inside-out）：** 头显相机观察手柄上的 **LED 灯环**。

```
头显相机图像 → 检测 LED 亮点 → PnP 求解位姿
```

| 优点 | 缺点 |
|------|------|
| 精度高、延迟低 | 灯环被手遮挡时失效 |
| 成熟 | 灯环增大手柄体积 |

**PnP 问题：** 已知 3D 灯点坐标和 2D 投影，求 $[R|t]$。

#### 3.3 IMU 融合

手柄内置 **IMU**（加速度计 + 陀螺仪）：

| 传感器 | 擅长 | 弱点 |
|--------|------|------|
| **陀螺仪** | 短期旋转精确 | 漂移 |
| **加速度计** | 重力方向 | 运动加速度干扰 |
| **光学** | 长期位置/姿态准 | 遮挡失效 |

**融合框架：**

| 类型 | 说明 |
|------|------|
| **松耦合** | 光学与 IMU 独立解算，后期加权融合 |
| **紧耦合** | EKF/ESKF 统一状态向量，联合优化 |
| **互补滤波** | 高频用陀螺，低频用光学校正 |

**状态向量（常见）：**

$$
\mathbf{x} = [p, v, q, b_g, b_a]^T
$$

位置、速度、四元数、陀螺 bias、加速度 bias。

#### 3.4 PICO Centaur 多模态方案（面试重点）

PICO 为**无灯环 / 小型化手柄**提出 **Centaur** 融合架构：

```
        ┌─────────────┐
        │  光学 LED   │──→ 高精度，怕遮挡
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │  IMU 惯性   │──→ 高帧率，短期准，长期漂
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │ CV 裸手图像  │──→ 手遮挡时估计手柄在手中的位姿
        └──────┬──────┘
               │
        ┌──────▼──────┐
        │ 神经网络融合 │──→ 端到端或多模态加权
        └──────┬──────┘
               ↓
           6DoF 输出
```

**核心思想：**
- 光学与裸手 CV **互补**：灯被挡时，从手部图像推断手柄 pose  
- 手柄与手有**物理约束**（握持关系）  
- IMU 在视觉失效的短窗口内桥接  
- **Down-Top** 端到端：用多目时序全局上下文一次预测 6DoF，检出率显著提升（公开资料：36% → 93%）

#### 3.5 融合算法面试答题模板

1. **各模态置信度：** $w_{opt}, w_{imu}, w_{cv}$，根据遮挡检测动态调整  
2. **光学失效检测：** LED 可见数量 < 阈值 → 降 $w_{opt}$  
3. **IMU 积分：** 两帧视觉之间用 gyro 传播旋转  
4. **CV 分支：** 手-手柄相对 pose 网络，输入多帧 crop  
5. **输出平滑：** 四元数 slerp，避免欧拉角万向节锁

## 三、面部追踪（Face Tracking）

> **完整 Pipeline（3DMM、TrueDepth、MediaPipe、Retargeting）：** [12-3D人脸表情追踪详解.md](./12-3D人脸表情追踪详解.md)  
> **眼动追踪（Gaze、Foveated Rendering）：** [13-眼动追踪详解.md](./13-眼动追踪详解.md)

### Q4：blendshape、实时性、表情驱动？

#### 4.1 面部追踪输出什么？

| 输出 | 用途 |
|------|------|
| **头部 6DoF** | 头像朝向 |
| **Blendshape 权重** | 52+ ARKit 标准表情系数 |
| **眼部 gaze** | 眼动、注视点渲染 |
| **嘴唇 / 下巴** | 语音口型同步（lip sync） |

#### 4.2 Blendshape 模型

将人脸 mesh 表示为**中性脸 + 各表情形变基的线性组合**：

$$
\mathbf{v} = \mathbf{v}_{neutral} + \sum_{i=1}^{N} w_i \mathbf{B}_i
$$

- $\mathbf{v}$：顶点坐标向量  
- $\mathbf{B}_i$：第 $i$ 个 blendshape 基（如 mouthSmileLeft）  
- $w_i \in [0, 1]$：网络回归的权重  

**ARKit 52 维：** eyeBlink, jawOpen, mouthSmile, browInnerUp 等 — 行业标准，方便跨平台 avatar。

#### 4.3 Pipeline

```
相机（前置 RGB / IR）→ 人脸检测 → 关键点（468 点 MediaPipe Face Mesh）
    → Blendshape 回归网络 → 52 维权重
    → 驱动 avatar / 社交应用
```

| 阶段 | 延迟 |
|------|------|
| 人脸检测 | 2–4 ms |
| Face mesh | 5–10 ms |
| Blendshape | 2–5 ms |
| **合计** | **10–20 ms**（可接受） |

#### 4.4 实时性挑战

| 挑战 | 对策 |
|------|------|
| 光照变化 | IR 结构光 / 均匀补光 |
| 眼镜 / 口罩 | 鲁棒训练数据；降级只追踪上半脸 |
| 快速表情 | 高帧率 + 低滞后滤波 |
| 身份一致性 | 个性化校准（用户中性脸扫描） |
| 与透视相机争用 | 独立 IR 相机或时间复用 |

#### 4.5 表情驱动应用（PICO 场景）

- **社交 VR：** avatar 表情同步  
- **会议纪要：** 表情辅助情绪分析（隐私敏感，需合规）  
- **眼动点渲染（Foveated Rendering）：** 眼球追踪降低 GPU 负载  

**面试要点：** Blendshape 是**低带宽**表情传输格式（52  float vs 完整 mesh）；权重需平滑避免表情抖动。

## 四、遮挡恢复

### Q5：手柄被手挡住时如何保持追踪？

这是 PICO Centaur 方案的**核心动机**，需系统性回答。

#### 5.1 问题分解

```
正常：相机看到 LED 灯环 → 光学解算 6DoF ✓
遮挡：手挡住灯环 → 光学失败 ✗
目标：仍输出连续、可用的 6DoF
```

**失败表现：** 手柄「飞走」、冻结、跳变 — 用户体验极差。

#### 5.2 分层恢复策略

```
Layer 1: IMU 短期桥接（50–200ms）
    陀螺积分维持旋转，加速度计辅助
    位置会漂，但旋转通常够用

Layer 2: CV 手-柄关系估计
    检测手 pose → 推断握持点 → 估计手柄相对手的 rigid transform
    利用「手几乎一直握着手柄」这一先验

Layer 3: 运动学约束
    手腕-手柄距离有界，相对旋转有限

Layer 4: 多目时序网络（Down-Top）
    多帧 + 多相机全局上下文，直接从图像回归 6DoF
    不依赖单帧 LED 检测

Layer 5: 重捕获（Re-acquisition）
    手移开瞬间，光学重新锁定 LED，与 IMU 预测对齐（避免跳变）
```

#### 5.3 手-柄几何约束

握持时手柄坐标系与手掌坐标系近似**刚体连接**：

$$
T_{world}^{controller} = T_{world}^{hand} \cdot T_{hand}^{controller}
$$

$T_{hand}^{controller}$ 可由产品外形 CAD 先验或用户标定获得。

**网络学习：** 输入手部 21 关键点 + 上一帧手柄 pose → 输出 $\Delta pose$ 或绝对 pose。

#### 5.4 置信度融合与平滑

| 模态 | 遮挡时置信度 |
|------|----------------|
| 光学 | 低 |
| IMU | 中（旋转高，位置低） |
| CV 手-柄 | 高（若手可见） |

**融合：**

$$
T_{out} = \frac{w_o T_o + w_i T_i + w_c T_c}{w_o + w_i + w_c}
$$

（旋转实际用四元数加权 slerp，非线性）

**平滑切换：** 光学→CV 切换时，用 100–200ms 交叉淡入避免跳变。

#### 5.5 评测指标

| 指标 | 说明 |
|------|------|
| **遮挡恢复时间** | 从全遮挡到 pose 误差 < 阈值的毫秒数 |
| **遮挡期间 RMSE** | 位置/角度误差 |
| **丢失率** | 完全失去追踪的帧占比 |
| **快速甩动场景** | 健身、节奏游戏 stress test |

PICO 公开测试：运动健身、音游等快速甩动场景下保持稳定追踪。

#### 5.6 面试系统设计题（完整答法）

**问：** 设计手柄遮挡时的追踪方案。

```
1. 需求：遮挡 500ms 内旋转误差 < 5°，位置误差 < 2cm
2. 传感器：头显相机 + 手柄 IMU + 可选手柄相机
3. 检测遮挡：LED 可见数 < 3 → occlusion flag
4. 切换：降光学权重，升 IMU+CV 权重
5. CV：多目时序网络，输入手+历史帧
6. 约束：MANO 手模型 + 握持点 CAD
7. 恢复：LED 重现时，EKF 重初始化，slerp 对齐
8. 评测：遮挡手套测试、快速挥手测试
```

## 五、跨主题：统一交互坐标系

所有追踪结果需转换到**头显世界坐标系**：

$$
T_{world}^{obj} = T_{world}^{head} \cdot T_{head}^{cam} \cdot T_{cam}^{obj}
$$

- $T_{world}^{head}$：SLAM/VIO 提供  
- 手、手柄、面部可在不同相机上解算，必须**时间戳对齐**  

**MTP 延迟：** 追踪结果使用的相机帧与显示帧之间的延迟必须最小化；常用「最新帧」或「预测到显示时刻」外推。
