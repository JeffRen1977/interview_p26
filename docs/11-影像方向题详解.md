## 岗位视角：影像工程师在 MR 里解决什么？

Imaging JD 不是单纯「调 ISP 参数」，而是对 **用户看到的每一帧现实画面** 负责：

```
Sensor 光子 → ISP/AI-ISP → 计算摄影增强 → 色彩/几何对齐 → 与虚拟内容合成 → 透镜显示 → 人眼
         ↑________________ Imaging 团队主战场 ________________↑
```

**四大场景题**分别对应：暗光、全链路画质、语义智能、透视缺陷校正。

## 一、低光增强（Low-Light Enhancement）

### Q1：多帧对齐 + 网络去噪 + tone mapping 怎么做？

#### 1.1 为什么 MR 低光特别难？

| 挑战 | 原因 |
|------|------|
| 噪声明显 | 为降延迟常缩短曝光，ISO 升高 |
| 运动模糊 | 用户转头 + 卷帘快门 |
| 动态范围大 | 窗外亮、室内暗同时入画 |
| 不能拖延迟 | 增强算法必须在 10–15 ms 内完成 |
| 不能有明显拖影 | 多帧融合必须运动补偿 |

用户通过透视看到的「暗、噪、糊」会直接归因于产品画质。

#### 1.2 整体 Pipeline

```
短曝光连拍 N 帧（N=2~8，受延迟约束）
    ↓
[帧间对齐]  全局运动（头部位姿）+ 局部光流（场景运动物体）
    ↓
[多帧融合]  加权平均 / 选优 / 学习式融合
    ↓
[AI 单帧/多帧去噪]  RAW 或 RGB 域网络
    ↓
[Tone Mapping]  线性 HDR → 显示亮度
    ↓
[Gamma / 色彩]  对齐 Display P3
    ↓
输出至 VST 合成
```

#### 1.3 多帧对齐（Alignment）

**全局对齐 — 用 SLAM/IMU 位姿：**

已知帧 $t$ 与 $t+1$ 相对位姿 $T_{t\to t+1}$，深度 $D_t$，将像素重投影：

$$
\pi_{t+1}^{-1} \left( D_t(u,v) \cdot K^{-1} \pi_t(u,v) \right)
$$

- $\pi$：投影函数  
- 适合**相机整体运动**（转头）  

**局部对齐 — 光流 / 块匹配：**

对动态物体、近景视差用 Dense Optical Flow（RAFT 轻量版、DIS）做 pixel-wise warp。

**对齐误差 → ghosting（重影）：**

| 策略 | 说明 |
|------|------|
| 运动检测 mask | 高运动区域只用单帧，不融合 |
| 曝光 bracket 减少 | N=2 比 N=8 ghost 风险低 |
| 鲁棒融合权重 | 不一致区域降权 |

**MR 约束：** 通常 **N≤4**；更多帧 → 更好 SNR，但延迟和对齐难度上升。

#### 1.4 网络去噪

**噪声模型（面试常问）：**

$$
I_{noisy} = I_{clean} + n_{shot} + n_{read}
$$

- shot noise：与信号相关，泊松  
- read noise：与信号无关，高斯  

**方案对比：**

| 方案 | 说明 | 端侧 |
|------|------|------|
| 传统 BM3D / NL-Means | 无训练，慢 | 难实时全分辨率 |
| 多帧时域 NR | 对齐后加权平均 | ✓ 常用 baseline |
| DnCNN / FFDNet | 单帧学习去噪 | ✓ 轻量版 |
| NAFNet / Restormer-S | 高质量，较重 | 需 INT8 + 降分辨率 |
| **RAW 域 AI-ISP** | 保留动态范围 | Imaging 岗加分 |

**训练数据：** 真实 sensor 噪声配对（短曝光 vs 长曝光 GT），或 **噪声合成**（Poisson-Gaussian）。

**损失：**

$$
\mathcal{L} = \| f_\theta(I_{noisy}) - I_{clean} \|_1 + \lambda \mathcal{L}_{perceptual}
$$

低光场景 perceptual/LPIPS 权重可适当提高，避免过度平滑。

#### 1.5 Tone Mapping

多帧融合后仍是**线性高动态范围**，需压缩到显示 8–10 bit。

| 方法 | 说明 |
|------|------|
| 全局 gamma | 快，易丢细节 |
| Reinhard / Drago | 经典 HDR TM |
| 局部 TM（双边域） | 细节好，可能 halo |
| **学习式 TM** | 网络直接输出 display-referred 图像 |

**关键：** 在**线性光域**完成融合和去噪，**最后一步**再做 tone mapping + gamma。

$$
L_{display} = \mathrm{TM}(L_{linear})
$$

#### 1.6 延迟预算示例（低光模块）

| 模块 | ms |
|------|-----|
| 双帧采集（pipeline 并行） | 0（与上一帧重叠） |
| IMU 位姿 warp | 2 |
| 光流（半分辨率） | 3 |
| 融合 + AI NR（INT8） | 6 |
| Tone mapping | 2 |
| **合计** | **~13** |

#### 1.7 评测

| 客观 | 主观 |
|------|------|
| PSNR/SSIM vs 参考长曝光 | MOS 清晰度、噪声、自然度 |
| LPIPS | 晕动症/舒适度（MR 特有） |
| 端到端延迟 | 暗光场景可用性 |

## 二、MR 显示画质（Capture-to-Display 全链路）

### Q2：如何优化 capture-to-display 全链路？

#### 2.1 什么是 Capture-to-Display（C2D）？

从传感器光子到用户视网膜的**完整成像链**，Imaging 团队需对最终主观画质负责，而非只优化某一个算法模块。

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ Sensor  │ → │   ISP   │ → │ 计算摄影 │ → │ 几何处理 │ → │ 显示前处理│ → │ Display │
│ 曝光3A  │   │ demosaic│   │ NR/HDR  │   │ 畸变校正 │   │ 透镜畸变 │   │ 面板    │
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
     ↑____________ 每段都有画质杠杆，需联合调优 ____________↑
```

#### 2.2 各段优化杠杆

| 阶段 | 画质杠杆 | 与谁协作 |
|------|----------|----------|
| **Sensor + 3A** | 曝光时间、增益、HDR 模式 | 相机硬件、驱动 |
| **ISP** | AWB 稳定性、demosaic、NR 强度 | 芯片 ISP 团队 |
| **计算摄影** | 去噪、HDR、锐化、超分 | 算法 |
| **几何** | 畸变、rolling shutter 校正 | SLAM、标定 |
| **色彩** | CCM、色域映射 P3 | 显示团队 |
| **显示** | 透镜畸变、色差、亮度均匀性 | 光学 |
| **合成** | 虚实色彩/亮度匹配 | 渲染引擎 |

#### 2.3 系统级优化思路（面试高分）

**1. 联合曝光策略（AE for MR）**

透视场景曝光目标不同于拍照：
- 不能只顾窗外亮部  
- 室内主体亮度要够  
- 曝光切换要**平滑**，避免亮度跳变  

**2. 跨模块 tone 一致性**

$$
L_{final} = f_{display}\big(f_{enhance}(f_{ISP}(RAW))\big)
$$

各 $f$ 需在同一色彩空间约定下标定，避免「ISP 已压暗、增强又拉亮」的打架。

**3. 分辨率与 FOV 匹配**

相机 FOV < 显示 FOV → 边缘黑边或拉伸  
→ 需光学设计 + 数字扩展（边缘质量下降是成像问题）

**4. MTP 与画质的权衡**

| 策略 | 效果 |
|------|------|
| 跳过增强 | 延迟低，画质差 |
| 降分辨率增强再上采样 | 平衡 |
| 注视点增强（foveated） | 注视区高质量 |

**5. 出厂 + 个体标定**

每台头显相机-显示标定参数写入 EEPROM，Imaging 负责标定流程与 QA。

#### 2.4 C2D 优化方法论

```
1. 建立 Golden 场景集（暗光、高对比、色彩丰富、运动）
2. 分段 profiling：每模块延迟 + PSNR + MOS
3. 找瓶颈：是噪声？色差？锐度？延迟？
4. A/B 单变量调优（一次只动一个模块）
5. 端到端 MOS 验收（非只看中间指标）
```

#### 2.5 关键指标 Dashboard

| 指标 | 目标方向 |
|------|----------|
| MTP 延迟 | ↓ < 20–25 ms |
| 全链路 PSNR/LPIPS | ↑ |
| ΔE 色差（vs 参考） | ↓ |
| 亮度均匀性 | 中心-边缘 < 阈值 |
| 锐度 MTF | 中频足够 |
| MOS | ↑ |
| 功耗 / 温升 | ↓ |

## 三、VLM 场景理解（端侧轻量 + 云端分流）

### Q3：端侧轻量分类 + 云端大模型分流怎么设计？

#### 3.1 业务需求

MR 用户可能问：「这是什么？」「我面前的桌子能放多大物体？」「有没有剪刀？」

| 需求类型 | 实时性 | 算力 | 适合位置 |
|----------|--------|------|----------|
| 场景类别（室内/户外/办公室） | 高 | 低 | 端侧 |
| 物体存在性（有桌子/有人） | 高 | 低 | 端侧 |
| 开放语义描述 | 低 | 高 | 云端 VLM |
| 复杂推理（数清几本书） | 低 | 高 | 云端 |

#### 3.2 端云协同架构

```
┌─────────────────────────────────────────────────────────┐
│                        端侧（每帧/每秒）                   │
│  相机 → ISP → 轻量 Backbone（MobileNet / SigLIP-Tiny）    │
│           → 场景分类头（室内/户外/暗光）                     │
│           → 物体 presence 检测（可选 YOLO-Nano）           │
│           → 触发器逻辑                                     │
└───────────────────────────┬─────────────────────────────┘
                            │ 触发时上传
                            ▼
┌─────────────────────────────────────────────────────────┐
│                        云端（异步）                        │
│  压缩图像 或 visual embedding（512–1024 维）              │
│       → VLM（Qwen-VL / GPT-4V / 自研）                    │
│       → 结构化 JSON 回复                                   │
└───────────────────────────┬─────────────────────────────┘
                            ▼
                     端侧 UI 展示 / MR 交互逻辑
```

#### 3.3 端侧轻量模块设计

**场景分类器：**

- 输入：256×256 RGB（或 ISP 后 YUV）  
- 模型：MobileNetV3 / EfficientNet-Lite，< 5 MB  
- 输出：{indoor, outdoor, lowlight, high_motion, ...}  
- 推理：< 3 ms INT8 on NPU  

**触发器逻辑（规则 + 小模型）：**

```python
if user_voice_query or button_press:
    trigger_cloud_vlm()
elif scene_change_detected():  # 分类结果变化 + 置信度
    update_local_context()
elif lowlight_classifier and enhancement_mode:
    switch_isp_profile("lowlight")
```

**Visual Embedding 上传（隐私友好）：**

- 端侧 SigLIP/CLIP image encoder → 512-d 向量  
- 云端只做 text-image 检索或 VLM conditioning  
- **不上传原图** — 面试隐私加分点  

#### 3.4 云端 VLM 交互

**请求：**

```json
{
  "embedding": [...],
  "query": "用户面前的桌面上有什么？",
  "context": {"scene": "indoor_office", "depth_plane": "desk_0.8m"}
}
```

**响应：**

```json
{
  "objects": ["laptop", "mug", "notebook"],
  "answer": "桌面上有笔记本电脑、杯子和笔记本。",
  "placement_hint": {"surface": "desk", "free_area_cm2": 1200}
}
```

#### 3.5 分流决策表

| 任务 | 端侧 | 云端 | 理由 |
|------|------|------|------|
| 暗光 ISP 参数切换 | ✓ | ✗ | 每帧、低延迟 |
| 场景分类 | ✓ | ✗ | 实时上下文 |
| 固定类别物体检测 | ✓ | ✗ | COCO 80 类够用 |
| 开放词汇问答 | 触发 | ✓ | 需大模型 |
| 精细分割「红色手柄螺丝刀」 | 触发 | ✓ | 开放语义 |
| 模型 OTA | 下载 | ✓ 托管 | 非实时 |

#### 3.6 延迟与体验

| 模式 | 用户感知 |
|------|----------|
| 端侧 only | 即时，能力有限 |
| 云端 only | 慢（RTT 100–500 ms），能力强 |
| **端侧先答 + 云端细化** | 最佳：先显示「办公室场景」，1s 后补充细节 |

#### 3.7 隐私与合规

- 用户可控开关：「允许场景理解上云」  
- 默认上传 embedding 而非原图  
- 本地缓存最近 N 次 embedding，支持离线重复查询  
- 敏感区域（人脸）mask 后再编码  

**Imaging 岗职责：** 提供**高质量、标准化的视觉输入**（ISP 稳定、色彩一致），使 VLM 输入分布稳定。

## 四、透视色差 / 模糊（Camera-Display Calibration）

### Q4：相机-显示标定与实时增强怎么做？

#### 4.1 问题来源

MR 透视中，用户同时感知「相机拍到的现实」和「虚拟渲染」，任何不一致都会破坏沉浸感。

| 缺陷 | 表现 | 原因 |
|------|------|------|
| **色差** | 现实偏黄/偏绿，虚拟偏冷 | 相机 AWB vs 显示色温不一致 |
| **色域不匹配** | 现实灰暗，虚拟艳丽 | 相机 sRGB 截断 vs P3 显示 |
| **模糊** | 透视画面发糊 | 对焦距离、运动模糊、低分辨率上采样 |
| **几何色差** | 边缘 RGB 分离 | 透镜横向色差（lateral CA） |
| **亮度不均** | 边缘暗 | 相机 vignetting + 显示透镜 |

#### 4.2 相机-显示联合标定

**目标：** 建立从 **相机 RGB** 到 **显示 RGB** 的映射，使现实画面与虚拟内容「看起来像同一个世界」。

**色彩标定流程（工厂/实验室）：**

```
1. 头显佩戴标准色卡（ColorChecker）于固定距离
2. 相机采集 RAW/RGB
3. 显示屏幕显示已知色块图案（用户眼位模拟）
4. 测量或仿真：相机记录 vs 人眼应感知
5. 求解 CCM + 1D LUT（per channel tone）
6. 写入设备校准文件
```

**数学模型（简化）：**

$$
\mathbf{c}_{display} = \mathbf{M}_{C2D} \cdot f_{LUT}(\mathbf{c}_{camera}) + \mathbf{b}
$$

- $\mathbf{M}_{C2D}$：3×3 色彩校正矩阵（相机→显示）  
- $f_{LUT}$：每通道非线性 LUT  
- 实际可用 3D LUT（17³）统一建模  

**个体差异：** 瞳距、佩戴位置 → 产线粗标定 + 可选用户 fine-tune。

#### 4.3 实时色彩增强

**AWB 锁定 vs 连续 AWB：**

| 模式 | MR 透视 |
|------|---------|
| 连续 AWB | 色温随场景跳变，虚拟内容难匹配 |
| **场景 AWB + 平滑** | 变化率限制，推荐 |
| **锁定 AWB** | 色彩稳，但极端光源不准 |

**虚拟-现实色彩匹配：**

```
渲染引擎使用与透视相同的 display color space（P3）
    +
透视链路输出经过统一 CCM 映射到 P3
    +
环境光估计（可选）调整虚拟物体光照
```

**环境光估计：** 从透视画面统计平均亮度和色温 → 调整虚拟 light probe。

#### 4.4 模糊校正

**来源分解：**

| 类型 | 对策 |
|------|------|
| **离焦模糊** | 固定对焦距离（1–3 m）优化 MR；或 per-scene 对焦 |
| **运动模糊** | 缩短曝光、去模糊网络、头动预测 |
| **光学模糊** | 镜头 MTF 反卷积（谨慎，噪声放大） |
| **分辨率不足** | 超分（Real-ESRGAN 轻量）、锐化 |
| **上采样模糊** | 更好的 resize kernel（Lanczos vs bilinear） |

**去模糊网络（端侧）：**

$$
\hat{I}_{sharp} = f_\theta(I_{blur})
$$

- 盲去模糊：核未知  
- MR 中运动方向与头转相关 → 可用 IMU 作为 blur kernel 先验  

**锐化 vs 去模糊：**

- 锐化：高频提升，不能恢复真实细节  
- 去模糊：尝试反演成像过程，计算贵  
- **MR 透视：** 轻度锐化 + 适度去模糊网络更稳妥  

#### 4.5 横向色差（Lateral CA）校正

不同波长折射率不同 → 边缘 R/B 相对 G 偏移。

**校正：** 按通道做轻微 scale + shift warp，参数由镜头标定。

$$
u_R' = u_R + \Delta x_R(r),\quad u_B' = u_B + \Delta x_B(r)
$$

$r$ 为到光学中心距离。

#### 4.6 实时 Pipeline 集成

```
相机 RAW/RGB
    ↓ AWB/CCM（用标定参数）
    ↓ 去噪（若低光）
    ↓ 去模糊 / 锐化（按运动强度自适应强度）
    ↓ 3D LUT（相机→显示色彩映射）
    ↓ 显示畸变校正
    ↓ 与虚拟帧合成
```

**自适应强度：**

```
if motion_score > θ_high:
    deblur_strength = low   # 避免 artifact
    sharpen_strength = low
elif lowlight:
    denoise_strength = high
    sharpen_strength = medium
```

#### 4.7 评测

| 指标 | 方法 |
|------|------|
| ΔE00 | 色卡各 patch 与参考色差 |
| MTF | 斜边法测锐度 |
| 虚实边界主观测试 | 用户看虚拟物体「贴」在现实上是否自然 |
| AB 测试 | 有/无 C2D 校准的 MOS 对比 |

## 五、四场景联合系统设计（Imaging 岗压轴题）

**题：** PICO MR 暗光办公室透视体验优化（涵盖四场景）。

**参考答法：**

```
1. 低光：双帧 IMU 对齐 + RAW NR（8ms）+ 局部 tone mapping
2. C2D：Profiling 显示链路 ΔE，统一 P3 CCM，MTP < 22ms
3. VLM：端侧 scene=indoor_office 切换 ISP profile；用户问「桌上有什么」触发云端 VLM
4. 色差/模糊：出厂 C2D 3D LUT；运动模糊时降去模糊强度；AWB 平滑防跳变
5. 评测：暗光 MOS + ΔE + 丢失帧率 + VLM 响应延迟
```
