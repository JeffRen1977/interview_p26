# 12 - 3D 人脸重建与表情追踪（Animoji / Avatar）

基于工业界 **Animoji / 数字人 Avatar** 技术路线，梳理从 2D 图像（或深度图）到 3D 虚拟角色驱动的完整 Pipeline。适用于 PICO / MR 头显**内视相机**、手机前置摄像头等场景。

> **GitHub 阅读：** 公式使用 `$...$` / `$$...$$` 格式。  
> 简要版：见 [09-追踪交互题详解.md](./09-追踪交互题详解.md) 第三节「面部追踪」。

---

## 一、核心目标

从手机/头显前置摄像头采集的 **2D RGB** 或 **红外深度图** 中，实时估计：

| 输出 | 说明 |
|------|------|
| **3D 几何** | 人脸 mesh 或 3DMM 顶点 |
| **位姿 Pose** | 6DoF（旋转 + 平移） |
| **表情** | Blendshape 系数 $\beta$（如 ARKit 52 维） |

并**无缝映射**到 3D 虚拟模型（狐狸、数字人、Avatar），实现嘴型、眼神与动画同步。

**工业界技术路线：**

```
参数化先验模型（3DMM）
    + 深度学习回归 / 几何跟踪
    + 实时 Retargeting 到 Avatar
    + Capture-to-Display 闭环（< 30 ms）
```

---

## 二、核心基石：3DMM（3D Morphable Model）

从 2D 视频直接重建任意 3D 人脸极其病态；工业界普遍引入**先验**——3DMM，即可数学控制的「标准 3D 人脸骨架」。

### 2.1 形状模型

任意人脸几何形状 $S$ 可表示为：

$$
S = S_0 + B_{shape} \cdot \alpha + B_{exp} \cdot \beta
$$

| 符号 | 含义 |
|------|------|
| $S_0$ | 平均人脸 3D 形状（基准 mesh，通常上万顶点） |
| $B_{shape}$ | 身份基（Identity Basis），PCA 主成分 |
| $\alpha$ | 身份系数，决定脸型胖瘦、骨骼结构（**你是谁**） |
| $B_{exp}$ | 表情基（Expression Basis / Blendshapes） |
| $\beta$ | 表情系数，决定笑、眨眼、撇嘴（**你此刻什么表情**） |

**Animoji 核心：** 每帧高频更新 $\beta$（52+ 维）；$\alpha$ 通常在首次录入或前几帧估计后**相对固定**。

### 2.2 纹理模型（扩展）

完整 3DMM 还可建模纹理：

$$
T = T_0 + B_{tex} \cdot \delta
$$

面试若追问 Avatar 肤色、化妆，可提及纹理系数 $\delta$。

### 2.3 ARKit 52 Blendshapes

苹果 ARKit 将 $\beta$ 规范为 **52 个语义维度**，便于跨引擎复用：

| 类别 | 示例系数 |
|------|----------|
| 眼部 | `eyeBlinkLeft`, `eyeBlinkRight`, `eyeLookUpLeft` |
| 眉部 | `browInnerUp`, `browDownLeft` |
| 嘴部 | `jawOpen`, `mouthSmileLeft`, `mouthPucker` |
| 脸颊 | `cheekPuff`, `cheekSquintLeft` |

每个系数 $\beta_i \in [0, 1]$，表示该表情维度的激活强度（如 `eyeBlinkLeft: 0.8`）。

**优势：** 低带宽（52 个 float vs 完整 mesh）、与 Unity/Unreal 标准兼容。

## 三、方案演进：深度几何 vs 纯 RGB 深度学习

### 3.1 方案 A：深度信息 + 物理几何（Apple TrueDepth 经典路线）

**硬件：** 结构光（点阵投影 + 红外接收），生成高质量实时深度图。

**流程：**

```
结构光投射红外点阵
    ↓
红外相机采集 → 实时 Depth Map / 3D 点云
    ↓
ICP 刚性对齐：3DMM mesh ↔ 点云，解算 6DoF 位姿 (R, t)
    ↓
非刚性变形：眼、嘴边缘区域精细拟合
    ↓
解算各 Blendshape 权重 β
```

#### ICP（Iterative Closest Point）

**目标：** 找变换 $T$ 使模型点云与观测点云距离最小：

$$
\min_{R,t} \sum_i \| R p_i + t - q_i \|^2
$$

- 交替：最近点对应 → SVD 求最优 $R,t$ → 迭代  
- **优点：** 几何可解释、深度已知时精度高、迭代可极快  
- **缺点：** 依赖深度硬件；纯 RGB 设备不可用  

**非刚性步骤：** 在 ICP 刚性对齐后，对眼睛、嘴唇等区域做 blendshape 系数优化或局部 deform。

---

### 3.2 方案 B：纯 RGB 端侧深度学习（现代主流）

随 NPU 算力提升，**单目 RGB**（安卓手机、MR 头显内视相机）即可达到优秀效果（如 MediaPipe Face Mesh、Google ML Kit）。

**流程：**

```
RGB 图像
    ↓
[1] 人脸检测 + 关键点（Landmark）
    468 / 478 个稠密 2D 点（眼睑、唇线、轮廓）
    ↓
[2] 端侧网络回归 3DMM 参数
    - 6DoF 位姿
    - 身份系数 α（首帧或注册时固定）
    - 表情系数 β（每帧更新）
    ↓
[3] 可选：3D mesh 顶点直接输出
    ↓
Retargeting → Avatar 渲染
```

#### 阶段 1：关键点检测

| 方案 | 点数 | 特点 |
|------|------|------|
| **MediaPipe Face Mesh** | 468 | 端侧成熟，唇眼精细 |
| **iPhone ARKit** | 1220+ mesh | 与 blendshape 一体 |
| **轻量 CNN** | 68 / 98 传统点 | 更快，精度略低 |

网络多为轻量 CNN 或 MobileNet backbone + heatmap / direct regression。

#### 阶段 2：参数回归

**输入：** 裁剪人脸 ROI +（可选）2D landmarks  
**输出：** $\alpha$, $\beta$, 6DoF pose  

**训练监督：**
- 3DMM 拟合伪标签  
- 多视角 3D 扫描数据  
- 合成渲染数据  

**端侧优化：** INT8 量化、NPU（SNPE/QNN、CoreML、TFLite GPU/NPU Delegate）。

#### 方案 A vs B 对比

| 维度 | 深度 + ICP | 纯 RGB 深度学习 |
|------|------------|-----------------|
| 硬件 | 结构光 / ToF | 普通 RGB 相机 |
| 深度精度 | 高 | 需网络估计或省略 |
| 暗光 | 红外相对稳 | 依赖 ISP / 补光 |
| 成本 | 高 | 低 |
| 代表 | iPhone TrueDepth Animoji | MediaPipe、PICO/Quest Avatar |
| MR 头显 | 部分机型有 IR | **内视 RGB 为主流** |

## 四、数据流与 Avatar 映射（Retargeting）

### 4.1 从系数到虚拟角色

算法在数毫秒内得到 52 维 $\beta$ 后：

```
β[0..51]  (eyeBlinkLeft=0.8, mouthSmileRight=0.5, ...)
    ↓
Retargeting（重定向）
    ↓
游戏引擎（Unity / Unreal）或自研渲染器
    ↓
Avatar mesh 的 52 个预制 Blendshape 目标
    ↓
线性插值形变 → 屏幕显示
```

**Blendshape 形变（逐顶点）：**

$$
\mathbf{v} = \mathbf{v}_{neutral} + \sum_{i=1}^{N} \beta_i \mathbf{B}_i
$$

与 3DMM 公式一致；Avatar 艺术家预制与 ARKit 语义对齐的 $\mathbf{B}_i$。

### 4.2 重定向（Retargeting）难点

| 问题 | 说明 |
|------|------|
| **拓扑不同** | 用户人脸 vs 狐狸/卡通角色 mesh 不同 |
| **语义映射** | ARKit 52 维 → 角色自有 rig |
| **比例** | 嘴大的人 vs 狐狸嘴型需缩放规则 |
| **解：** | 线性映射表、艺术家调权重、或中间标准脸中转 |

### 4.3 Capture-to-Display 闭环

| 指标 | 目标 |
|------|------|
| 端到端延迟 | **< 30 ms**（≥ 30 FPS） |
| 唇形同步 | 语音通话场景 < 50 ms 仍可用 |
| 抖动 | 系数时间平滑，无「面部抽搐」 |

```
相机采集 (t)
    → 检测+回归 (t+8ms)
    → β 滤波 (t+10ms)
    → 渲染提交 (t+15ms)
    → 显示 (t+20~25ms)
```

**MTP 过大：** 用户感觉「嘴型慢半拍」，社交 VR 体验崩坏。

## 五、端侧落地：工程挑战与 Trade-offs

### 5.1 鲁棒性（遮挡与极端光照）

| 挑战 | 现象 | 对策 |
|------|------|------|
| 眼镜 | 反光、遮挡眼周 | 眼镜检测 + 降权眼周系数 |
| 刘海 | 眉、额关键点丢失 | 时序预测 + 遮挡 mask |
| 单侧强光 | 阴阳脸，关键点漂 | ISP 人脸曝光 + 数据增强训练 |
| 快速表情 | 单帧回归滞后 | 高帧率 + 轻量 temporal 模型 |

**时序滤波：**

| 方法 | 说明 |
|------|------|
| **一阶低通** | 简单，引入滞后 |
| **Kalman Filter** | 状态 = $\beta$ + 速度，预测+更新 |
| **One-Euro Filter** | 静止平滑、运动跟手 |
| **轻量 Temporal Transformer / RNN** | 利用前后帧上下文，消抖 |

**状态机：**

```
if face_confidence < θ:
    hold_last_β or fade_to_neutral
else:
    update β with filter
```

### 5.2 异构计算（功耗与发热）

人脸追踪常为**常驻后台**任务，需控制功耗避免降频。

| 模块 | 推荐硬件 |
|------|----------|
| 人脸检测 + landmark CNN | **NPU**（SNPE/QNN、CoreML、NNAPI） |
| 3DMM 参数回归 | **NPU** INT8 |
| ICP / 几何优化（若有深度） | **DSP / CPU** 短循环 |
| Blendshape 应用 + 蒙皮 | **GPU** |
| 滤波与逻辑 | **CPU** |

**高通平台示例：**

```
RGB → SNPE/QNN 运行 face mesh 模型
    → DSP 做轻量后处理
    → Unity 插件读 β 驱动 Avatar
```

**优化手段：**
- INT8 PTQ（见 [07-端侧部署题详解.md](./07-端侧部署题详解.md)）  
- 隔帧检测 + 中间帧跟踪  
- 降低输入分辨率（192×192 ROI）  
- 多人脸时只跟踪主用户  

### 5.3 隐私与 MR 场景

| 点 | 做法 |
|----|------|
| 原始图像 | 不出设备，仅本地推理 |
| 上传社交 | 只传 $\beta$ 系数或动画流，不传 mesh |
| 眼动 | 更敏感，需单独授权 |

## 七、面试常问题

### Q1：为什么用 3DMM 而不是直接回归 mesh 顶点？

| 直接回归顶点 | 3DMM |
|--------------|------|
| 输出维度极高（×3 万维） | 低维 $\alpha,\beta$（~50–300 维） |
| 易不合理拓扑 | 先验保证人脸流形 |
| 难泛化 | 数据效率高 |

### Q2：$\alpha$ 和 $\beta$ 分别什么时候更新？

- $\beta$：**每帧**更新（实时表情）  
- $\alpha$：注册/首几秒估计后**固定**或极低频更新（身份）  

### Q3：ICP 和深度学习如何选？

- 有可靠深度 → ICP 做 pose，网络做 expression  
- 仅 RGB → 端到端网络或 2D landmark + 3DMM fitting  

### Q4：如何评估？

| 类型 | 指标 |
|------|------|
| 几何 | 3D 顶点误差（有 3D GT 时） |
| 2D | landmark NME（归一化平均误差） |
| 表情 | $\beta$ 与 GT 相关系数 |
| 主观 | MOS 自然度、唇同步满意度 |
| 系统 | 延迟、功耗、丢脸率 |
