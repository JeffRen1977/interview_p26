## 一、量化（Quantization）

### Q1：PTQ vs QAT 有什么区别？什么时候用哪个？

#### 基本概念

**量化：** 把 FP32 权重/激活映射到低比特（INT8/INT4/FP16），减少模型体积、内存带宽和计算量。

$$
q = \mathrm{clip}\left(\mathrm{round}\left(\frac{x}{s}\right) + z\right)
$$

- $s$：scale（缩放因子）
- $z$：zero point（零点偏移，非对称量化时用）

#### PTQ（Post-Training Quantization，训练后量化）

| 项目 | 说明 |
|------|------|
| **流程** | 训练好 FP32 模型 → 用校准集统计激活范围 → 直接转 INT8 |
| **优点** | 无需重新训练，速度快，工程成本低 |
| **缺点** | 精度损失可能较大，对激活分布敏感的模型掉点明显 |
| **常用校准** | MinMax、Moving Average、Entropy（KL 散度最小化）、Percentile |

**典型流程：**

```
FP32 模型 → 导出 ONNX → 校准集前向（100~500 张）→ 统计每层 scale
         → TensorRT/NCNN INT8 引擎 → 精度验证 → 部署
```

#### QAT（Quantization-Aware Training，量化感知训练）

| 项目 | 说明 |
|------|------|
| **流程** | 训练/微调时**模拟**量化（fake quant），让网络适应量化误差 |
| **优点** | 精度明显优于 PTQ，INT8 可接近 FP32 |
| **缺点** | 需要训练资源和时间，实现复杂 |
| **工具** | PyTorch `torch.quantization`、TensorRT QAT、MQBench |

**Fake Quant 前向：**

$$
\hat{x} = s \cdot \mathrm{round}\left(\mathrm{clip}\left(\frac{x}{s}, -128, 127\right)\right)
$$

前向用量化值，反向用 straight-through estimator（STE）传梯度。

#### 选型决策（面试直接背）

| 场景 | 推荐 |
|------|------|
| 快速验证、精度容忍度高 | PTQ |
| INT8 掉点 > 1–2% mAP / PSNR 不可接受 | QAT |
| 有预训练大模型、只需部署 | PTQ + 混合精度 |
| 量产前最终优化 | QAT 或 PTQ + 逐层敏感度分析 |
| 时间紧、无训练集群 | PTQ + FP16 敏感层 |

**PICO 口述模板：**

> 我们先用 PTQ 快速出 baseline，若关键指标不达标，对敏感层保留 FP16，其余 INT8；仍不够则对 head 或最后一层 block 做 QAT 微调 5–10 epoch。

### Q2：校准集（Calibration Set）怎么选？

校准集用于 PTQ 时估计各层激活的 dynamic range（min/max 或直方图）。

#### 原则

| 原则 | 说明 |
|------|------|
| **代表性** | 覆盖真实部署场景的光照、场景、运动模糊、遮挡 |
| **多样性** | 不同曝光、色彩、纹理、距离，避免单一分布 |
| **数量** | 通常 **100–500 张**足够；过多收益递减 |
| **与训练集关系** | 不必与训练集相同，但必须与**推理分布**一致 |
| **避免** | 全黑/全白极端图、过度重复场景 |

#### 实操建议

1. 从**量产设备**实际采集或仿真采集  
2. 按场景分层抽样：室内 40%、室外 30%、低光 20%、运动模糊 10%  
3. 包含 bad case（量化掉点严重的样本）  
4. 校准后必须在**独立验证集**上测精度，不能只看校准集  

#### 面试加分

- **per-channel vs per-tensor：** 卷积权重常用 per-channel 量化，精度更好  
- **校准方法：** Entropy calibration（TensorRT 默认）通常优于简单 MinMax  
- **序列模型：** 视频/时序模型需包含运动序列，而非单帧

### Q3：哪些层对量化敏感？怎么处理？

#### 高敏感层（通常保留 FP16 或更高精度）

| 层类型 | 原因 |
|--------|------|
| **第一层 Conv** | 输入动态范围大（RAW/RGB 像素值），统计不稳定 |
| **最后一层 Conv / Head** | 直接输出检测/分割/回归，误差放大 |
| **小 channel 数层** | 量化噪声占比大 |
| **Sigmoid/Softmax 前一层** | 概率校准敏感，阈值附近易翻转 |
| **Depthwise Conv** | 通道间不共享 scale，易累积误差 |
| **上采样 + 相加（skip）** | 多分支融合，量化误差叠加 |
| **Attention / LayerNorm** | 动态范围大，INT8 困难 |

#### 低敏感层

- 中间大 channel 的 3×3 Conv、ReLU 堆叠  
- 深层语义特征提取 block  

#### 补救策略（由易到难）

```
1. 混合精度：敏感层 FP16，其余 INT8
2. 换 per-channel 量化
3. 换 Entropy / Percentile 校准
4. 扩大/优化校准集
5. QAT 微调
6. 知识蒸馏：FP32 teacher → INT8 student
```

#### 敏感度分析工具

- TensorRT `polygraphy` / layer-wise precision override  
- NCNN `quantizemodel` 逐层对比  
- 手动：逐层 FP16、其余 INT8 二分定位掉点层  

**面试一句话：** 首尾层 + head + 小 channel + 激活前的层最敏感；用混合精度是最快的工程解法。

## 二、压缩（Compression）

### Q4：剪枝、蒸馏、通道剪枝的区别？如何组合？

#### 三种方法对比

| 方法 | 压缩对象 | 原理 | 典型收益 | 代价 |
|------|----------|------|----------|------|
| **剪枝（Pruning）** | 权重/结构 | 去掉不重要连接或通道 | 稀疏加速（需硬件支持） | 需微调恢复精度 |
| **通道剪枝（Channel Pruning）** | 整个 filter/channel | 结构化剪枝，物理删除通道 | 算力/内存线性下降，易部署 | 需重训或 fine-tune |
| **蒸馏（Distillation）** | 知识迁移 | 大模型教小模型 | 小模型精度更高 | 需要 teacher 推理 |

#### 非结构化剪枝 vs 结构化（通道）剪枝

**非结构化：** 去掉个别权重 → 稀疏矩阵 → 需要专用稀疏加速（多数移动端 NPU **不支持**或收益有限）

**结构化（通道剪枝）：** 去掉整个 filter → 输出 channel 减少 → 下一层 input channel 同步减少 → **标准 GEMM/Conv 直接加速**

**面试重点：** 端侧部署优先 **结构化通道剪枝**，非结构化剪枝在 XR 设备上往往不划算。

#### 知识蒸馏

**Soft label 损失：**

$$
\mathcal{L} = \alpha \mathcal{L}_{hard}(y, \hat{y}) + (1-\alpha) \mathcal{L}_{soft}(p_s, p_t)
$$

$$
\mathcal{L}_{soft} = \mathrm{KL}\left(p_s^{1/T} \| p_t^{1/T}\right)
$$

- $p_t$：teacher 的 softmax 输出（暗知识）  
- $T$：温度，越大分布越软  

**特征蒸馏：** 对齐中间层 feature map（FitNets、AT 等）

#### 组合策略（推荐 pipeline）

```
Step 1: 蒸馏 — 大 FP32 teacher → 小 FP32 student（保精度）
Step 2: 通道剪枝 — 去掉冗余 channel，fine-tune 恢复
Step 3: 量化 — PTQ/QAT 转 INT8
Step 4: 部署 — TensorRT/NCNN 推理
```

**为什么这个顺序：**
- 先蒸馏得到小模型，再剪枝进一步减结构  
- 剪枝后再量化，比直接量化大模型掉点少  
- 剪枝+量化叠加可实现 **4–8×** 加速  

#### PICO 场景举例

> 手部关键点模型：MobileNet teacher → 定制轻量 student（蒸馏）→ 通道从 256 剪到 128 → INT8 PTQ → NCNN 在骁龙 XR2 上 8ms 推理。

## 三、推理框架选型

### Q5：TensorRT / NCNN / MNN / ONNX Runtime 怎么选？

#### 框架对比总表

| 框架 | 开发商 | 主要平台 | 优势 | 劣势 |
|------|--------|----------|------|------|
| **TensorRT** | NVIDIA | NVIDIA GPU / Jetson / 部分 ARM | 极致优化、INT8/FP16、layer fusion 强 | 绑定 NVIDIA 生态，VR 头显多用骁龙 |
| **NCNN** | 腾讯 | ARM CPU / Vulkan GPU | 移动端成熟、无第三方依赖、Vulkan 加速 | 大模型/复杂 op 支持需自己改 |
| **MNN** | 阿里 | ARM CPU / GPU / NPU | 多后端、阿里系优化好、支持训练 | 文档/community 相对 NCNN 少 |
| **ONNX Runtime** | Microsoft | 跨平台 CPU/GPU/NPU | 通用、op 覆盖广、易接入 | 峰值性能通常不如专用引擎 |
| **TFLite** | Google | ARM / Edge TPU | Android 生态好 | 复杂 CV 模型转换坑多 |
| **SNPE** | Qualcomm | 骁龙 NPU/DSP | XR 设备（PICO 多用骁龙）NPU 加速 | 高通生态绑定，调试门槛高 |

#### PICO / XR 设备选型建议

PICO 头显普遍基于 **高通骁龙 XR** 平台：

```
首选路径：
  PyTorch → ONNX → NCNN（CPU/Vulkan）或 SNPE（NPU/DSP）

备选路径：
  PyTorch → ONNX → ONNX Runtime（快速验证）
  PyTorch → ONNX → TensorRT（若在 Jetson / PC 仿真验证）

Android 侧：
  MNN / TFLite / NCNN 均可，看团队基建
```

#### 选型决策树（面试口述）

```
1. 目标芯片是什么？
   ├─ 骁龙 XR → NCNN + SNPE（吃满 NPU）
   ├─ NVIDIA → TensorRT
   └─ 跨平台验证 → ONNX Runtime

2. 延迟要求？
   ├─ <10ms → 必须 NPU/DSP + INT8
   └─ <30ms → CPU INT8 或 Vulkan 可能够

3. 模型复杂度？
   ├─ 标准 CNN → NCNN/MNN 成熟
   └─ Transformer/VLM → ONNX Runtime / 定制 op

4. 团队熟悉度 & 已有基建 → 优先已有栈
```

#### 各框架核心能力一句话

| 框架 | 一句话 |
|------|--------|
| TensorRT | NVIDIA 平台最强，builder 自动 fusion + INT8 |
| NCNN | 轻量 ARM 首选，PICO/手机 CV 部署常见 |
| MNN | 阿里出品，端侧全能，适合多业务线统一 |
| ONNX Runtime | 通用中间层，原型验证快，极致性能需 Execution Provider |

## 四、推理优化

### Q6：算子融合、内存复用、Pipeline 并行是什么？

#### 算子融合（Operator Fusion）

**问题：** 每层 Conv/BN/ReLU 单独执行 → 多次读写 DDR → 带宽瓶颈

**融合示例：**

```
未融合：Conv → 写回内存 → BN → 写回 → ReLU → 写回   （3 次 DRAM 访问）
融合后：Conv+BN+ReLU 一个 kernel，中间结果在寄存器/shared memory   （1 次写回）
```

**常见融合模式：**

| 融合 | 说明 |
|------|------|
| Conv + BN + ReLU | 最经典 |
| Conv + Add（残差） | ResNet 块 |
| MatMul + Add + ReLU | FC 层 |
| Depthwise + Pointwise | MobileNet 块 |

**谁来做：** TensorRT/NCNN/MNN 编译时自动 pattern match 融合；手写 CUDA/Vulkan shader 可更深优化。

**面试要点：** 融合减少的是 **内存带宽**，不是理论 FLOPs；端侧 often memory-bound。

---

#### 内存复用（Memory Reuse / In-place）

**问题：** 移动端内存 2–4GB，模型 + 中间 feature map 易 OOM

**策略：**

| 策略 | 说明 |
|------|------|
| **In-place 算子** | ReLU、Add 等原地操作，不分配新 buffer |
| **静态内存规划** | 编译期分析 tensor 生命周期，复用同一块内存 |
| **Arena 分配器** | 预分配推理 buffer，避免 malloc 碎片 |
| **降低 feature 精度** | 中间层 FP16，输入输出按需 |
| **减少 batch** | 端侧恒为 batch=1 |

**生命周期复用示意：**

```
Tensor A (layer 1 输出) ──→ layer 2 用后不再需要
                              ↓
Tensor B (layer 5 输入)  可复用 A 的内存地址
```

NCNN 的 `blob allocator`、TensorRT 的 `workspace` 都是这个思想。

---

#### Pipeline 并行（Pipeline Parallelism）

**问题：** 相机采集、预处理、推理、后处理、渲染串行 → 总延迟 = 各阶段之和

**优化：** 多阶段**流水线重叠**， akin CPU 流水线

```
无流水线（串行）：
|--采集--|--预处理--|--推理--|--后处理--|--渲染--|  总延迟 = 总和

流水线（双缓冲）：
帧 N:   |--采集--|--预处理--|--推理--|--后处理--|--渲染--|
帧 N+1:          |--采集--|--预处理--|--推理--|--后处理--|
                              ↑ 各模块同时工作不同帧
端到端吞吐提升，单帧延迟仍取决于最慢阶段
```

**端侧常见拆分：**

| 线程/模块 | 任务 |
|-----------|------|
| Thread 1 | 相机采集 + ISP 预处理 |
| Thread 2 | 模型推理（NPU/GPU） |
| Thread 3 | 后处理（NMS、滤波）+ 提交渲染 |

**注意：**
- Pipeline 降低的是**吞吐**和**排队等待**，单帧 **MTP 延迟**仍要优化关键路径  
- 双缓冲/三缓冲避免推理时阻塞采集  
- XR 对 **motion-to-photon** 极敏感，pipeline 不能引入额外 1 帧延迟（除非 acceptable）

**PICO 口述：**

> 我们把采集和推理放在不同线程，用无锁队列传递预处理后的 tensor；推理用 NPU 异步提交，CPU 同时准备下一帧；后处理和渲染合批，整体吞吐从 25fps 提到 45fps，单帧关键路径仍控制在 18ms 内。

## 五、系统权衡

### Q7：FP16 vs INT8 怎么选？

| 维度 | FP16 | INT8 |
|------|------|------|
| **精度** | 接近 FP32，掉点极小 | 可能掉 0.5–3%（视模型而定） |
| **速度** | 约 1.5–2× vs FP32 | 约 2–4× vs FP32 |
| **内存** | 一半于 FP32 | 四分之一于 FP32 |
| **硬件支持** | 广泛（GPU/NPU 均支持） | 需 INT8 加速单元 |
| **实现难度** | 低，几乎无损 | 需校准/QAT |

#### 决策建议

```
默认策略：
  Backbone / 主体网络 → INT8
  敏感层（首层、head、检测输出）→ FP16
  最终 lossy 环节（超分、去噪输出）→ 可对比 PSNR 后决定

若 INT8 掉点不可接受 → 全 FP16（仍比 FP32 快且省内存）
若算力仍不够 → INT8 + 通道剪枝 + 更小输入分辨率
```

**混合精度（MP）：** TensorRT 的 `precision_constraints` 可逐层指定 FP16/INT8。

### Q8：ARM CPU vs NPU 任务怎么划分？

#### 硬件角色

| 单元 | 特点 | 适合任务 |
|------|------|----------|
| **ARM CPU** | 灵活、延迟低启动、单线程弱 | 预处理、后处理、逻辑控制、小模型 |
| **GPU (Vulkan/OpenCL)** | 并行高、适合大 tensor | 中等 CNN、图像 resize、渲染 |
| **NPU/DSP (SNPE)** | 专用矩阵加速、能效高 | 主流 CNN/Transformer 推理 |
| **ISP（硬件）** | 固定 pipeline、极低功耗 | demosaic、NR、AE/AWB |

#### 任务划分原则

```
ISP（硬件）     ： RAW → YUV，能不走 CPU 就不走
CPU            ： resize、normalize、ROI 裁剪、时间戳对齐、NMS、跟踪滤波
NPU            ： 主推理网络（检测、分割、深度、增强）
GPU/Vulkan     ： 大分辨率 warp、渲染合成、部分 NCNN Vulkan 推理
```

#### 决策因素

| 因素 | 倾向 |
|------|------|
| 算子 NPU 是否支持 | 不支持则 CPU/GPU fallback |
| 数据是否在 NPU 内存 | 频繁 CPU↔NPU 拷贝抵消收益 |
| 模型大小 | 小模型（<5MB）CPU INT8 可能够快，不必上 NPU |
| 功耗预算 | 持续 MR 模式优先 NPU（能效比高） |
| 延迟 | NPU 有提交开销，极小模型 CPU 反而更快 |

#### 反面教训（面试加分）

- **频繁拷贝：** CPU 预处理 → NPU 推理 → CPU NMS，拷贝占 30% 时间 → 用 zero-copy / 统一内存  
- **NPU 冷启动：** 首次推理慢 → 启动时 warmup  
- **算子 fallback：** 一个不支持 op 整网回 CPU → 提前用 Netron 检查，改模型结构

## 六、端到端部署 Checklist（PICO 面试可主动提）

```
□ 模型：蒸馏/剪枝后的轻量结构
□ 量化：PTQ baseline → 敏感层 FP16 → 必要时 QAT
□ 转换：PyTorch → ONNX → 目标引擎（NCNN/SNPE）
□ 校准集：100+ 张，覆盖真实场景
□ 精度：客观指标 + 端侧 bad case
□ 性能：单帧延迟 p50/p99、内存峰值、功耗、温升
□ 流水线：采集/推理/后处理并行，避免阻塞
□ 异常：模型加载失败、推理超时、降级的 fallback
□ 迭代：OTA 模型更新、A/B 对比
```

## 七、速记表（考前 5 分钟）

| 主题 | 一句话 |
|------|--------|
| PTQ | 训好后量化，快但可能掉点 |
| QAT | 训练中模拟量化，精度好但费时 |
| 校准集 | 100–500 张，覆盖真实场景分布 |
| 敏感层 | 首层、末层、head、小 channel → FP16 |
| 通道剪枝 | 结构化，端侧可加速；优于非结构化 |
| 蒸馏 | 大模型教小模型，保精度再压缩 |
| 组合 | 蒸馏 → 剪枝 → 量化 → 部署 |
| NCNN | ARM/XR 常用，轻量 Vulkan |
| TensorRT | NVIDIA 平台首选 |
| ONNX Runtime | 跨平台验证快 |
| 算子融合 | 减 DRAM 访问，memory-bound 关键 |
| 内存复用 | 生命周期分析，in-place / arena |
| Pipeline | 多线程重叠阶段，提吞吐 |
| FP16 vs INT8 | 混合精度：主体 INT8 + 敏感 FP16 |
| ARM vs NPU | 前处理/NMS 在 CPU，主网在 NPU |

## 八、与项目结合的回答示例

**面试官：** 你做过端侧部署吗？具体怎么做的？

**参考回答：**

> 我在 `[项目]` 里把 `[模型]` 部署到 `[骁龙 XR2 / Android]` 上。先用 ONNX 导出，再用 NCNN 转 INT8，校准集用了 `[N]` 张真实场景图。首层和检测 head 保留 FP16，其余 INT8，mAP 从 `[X]` 降到 `[Y]`（掉 `[Z]` 个点）。通过 Conv+BN+ReLU 融合和 blob 内存复用，推理从 `[A]` ms 优化到 `[B]` ms。采集和推理分线程流水线，端到端延迟控制在 `[C]` ms 内，满足 `[VR 透视 / 手部追踪]` 的实时要求。
