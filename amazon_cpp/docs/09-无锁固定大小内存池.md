# Part 19 — 高性能数据面：无锁固定大小内存池

> 白板简化版（主记这个）：[`examples/11_lock_free_fixed_pool.cpp`](../examples/11_lock_free_fixed_pool.cpp)
>
> 示例刻意保留经典 Treiber free list，方便面试口述主线。本文后面章节是追问加分项：ABA、tagged index、永久 metadata、Per-Core Cache。

面向 AWS EC2 Nitro / Annapurna / DPDK 风格数据面。面试先写出四件事：

1. 热路径 **零动态内存分配**（调用方给连续 buffer）；
2. Free list + `CAS` 做无锁 `allocate` / `deallocate`；
3. Placement New + 显式析构；
4. Cache Line 对齐，减少 False Sharing。

然后主动说局限：裸指针 free list 有 ABA；全局 `head` 高竞争要用 Per-Core Cache。

---

## 一、题目

设计一个固定大小对象池：

- 调用方传入一块预分配连续内存；
- 构造时切成固定槽位；
- `allocate()` 无锁取出槽位，并用 Placement New 构造 `T`；
- `deallocate()` 显式析构 `T`，再把槽位放回空闲表；
- 池耗尽立即返回 `nullptr`，不在 Datapath 阻塞；
- 多生产者、多消费者并发安全；
- 槽位按 Cache Line 对齐。

典型用途不是分配任意大小内存，而是复用固定布局描述符：

```cpp
struct PacketDescriptor {
    uint64_t dma_address;
    uint32_t length;
    uint32_t queue_id;
};
```

## 二、基础结构

### 2.1 外部内存

池不拥有原始内存：

```text
raw_memory（调用方提供，要求对齐）
┌──────────── Slot 0（64B 对齐）────────────┐
│ permanent metadata │ placement-new T      │
├──────────── Slot 1（64B 对齐）────────────┤
│ permanent metadata │ placement-new T      │
├────────────────────────────────────────────┤
│ ...                                        │
└────────────────────────────────────────────┘
```

原始内存可以来自：

- `alignas(64)` 静态缓冲区；
- `mmap()`；
- `mmap(MAP_HUGETLB)` / hugetlbfs；
- NUMA-aware allocator；
- DPDK hugepage memzone。

**严谨表述：** “连续虚拟内存”不等于“连续物理内存”；普通 `mmap` 也不等于大页。是否物理连续、是否使用 HugeTLB，由底层分配方式决定，内存池本身不能保证。

### 2.2 为什么用索引而不是裸指针

固定池中的槽位可由 32-bit index 标识。全局头打包成一个 64-bit 原子：

```text
63                         32 31                          0
┌────────────────────────────┬────────────────────────────┐
│ version/tag（每次 CAS +1） │ slot index                 │
└────────────────────────────┴────────────────────────────┘
```

优势：

- 64-bit CAS 在主流 x86-64 / ARM64 通常是 lock-free；
- 不依赖把版本号塞进指针空闲位的非可移植技巧；
- 地址由 `base + index * block_size` 计算；
- tag 能识别 `A → B → A` 的头指针变化。

## 三、为什么“裸 `atomic<Node*>` Treiber Free List”并不安全

常见简化代码：

```cpp
Node* old_head = head.load();
while (old_head &&
       !head.compare_exchange_weak(old_head, old_head->next)) {
}
```

### 3.1 真实 ABA 反例

初始：

```text
head → A → B → C
```

线程 1：

```text
读取 old_head=A、next=B，然后暂停
```

线程 2：

```text
pop A
pop B
push A
```

此时：

```text
head → A → C
```

线程 1 的 CAS 仍看到 `head == A`，于是错误地写入之前保存的 `B`：

```text
head → B       // B 可能已作为活对象，C 也从链表中丢失
```

所以“上层严格控制生命周期，ABA 危害较低”不是正确结论。只要同一个节点能被快速 pop/push，ABA 就真实存在。

### 3.2 仅加 tag 还不够：节点生命周期竞态

线程 1 读取 `head=A` 后暂停；线程 2 pop A 并在 A 的内存上 Placement New 一个 `T`。线程 1 恢复后读取 `A->next`：

- `Node` 生命周期可能已经结束；
- 同一内存现在是 `T`；
- 读 `next` 可能与构造 `T` 并发，形成 data race。

因此修正版把 `next` 放在**永久存在的槽位元数据**里，不与 `T` 覆盖，并把它做成 atomic。即使某线程输掉 head CAS，它也不会读取已经变成另一种对象的内存。

## 四、修正版算法

完整实现见 [`11_lock_free_fixed_pool.cpp`](../examples/11_lock_free_fixed_pool.cpp)，核心流程如下。

### 4.1 Pop / Allocate

```cpp
old = head.load(acquire);
while (true) {
    index = index_of(old);
    if (index == null) return nullptr;

    next = slot[index].next.load(relaxed);
    desired = {old.tag + 1, next};
    if (head.CAS_weak(old, desired, acquire, acquire)) break;
}

placement_new(object_address(index), T(args...));
```

成功 CAS 的线程独占该槽位。构造函数若抛异常，必须把槽位重新放回池，否则容量永久减少。

### 4.2 Push / Deallocate

```cpp
object->~T();
old = head.load(relaxed);
do {
    slot[index].next.store(index_of(old), relaxed);
    desired = {old.tag + 1, index};
} while (!head.CAS_weak(old, desired, release, relaxed));
```

`release` 发布：

- `T` 已析构；
- `next` 已写好；
- 槽位状态已恢复为空闲。

后续成功 pop 的 `acquire` 能看到这些写入。

### 4.3 为什么 `weak`

`compare_exchange_weak` 允许伪失败。这里本来就在循环里：

- 真竞争失败要重试；
- LL/SC 平台伪失败也重试；
- 无需 `strong` 替调用方隐藏重试。

不能笼统说 `strong` “内部一定包含循环”；标准只规定可观察语义，不规定具体实现。面试应说：

> 单次、不能容忍伪失败时用 strong；显式 CAS 循环中通常用 weak。

## 五、内存序

| 操作 | 内存序 | 原因 |
|------|--------|------|
| Push 写 `next` | `relaxed` | 由后续 head 的 release CAS 统一发布 |
| Push CAS 成功 | `release` | 发布槽位已可复用 |
| Push CAS 失败 | `relaxed` | 只需取得新 head 后重算 |
| Pop head load/CAS 成功 | `acquire` | 与 push release 配对，看到完整元数据 |
| Pop CAS 失败 | `acquire` | `expected` 被更新后，下一轮会据此读 `next` |

`head` 只负责所有权转移，不需要每处都用 `seq_cst`。

## 六、Placement New 与显式析构

```cpp
T* object = ::new (address) T(args...);
```

Placement New：

- **不申请内存**；
- 只在已有地址上启动 `T` 的生命周期并调用构造函数；
- 返回该地址上的 `T*`。

```cpp
object->~T();
```

显式析构：

- 结束 `T` 生命周期；
- 释放 `T` 自身管理的外部资源；
- **不释放槽位存储**。

绝不能：

```cpp
delete object;  // 错：存储不来自普通 new
```

还要修正一个常见误解：普通 `malloc/new` 并非每次都进入内核，用户态 allocator 通常有 arena/cache。对象池消除的是 allocator 元数据、锁竞争、碎片和不可预测延迟，而不是简单地“每次省一个系统调用”。

## 七、Cache Line 与扩展性

### 7.1 64B 对齐解决什么

若两个核心分别写相邻对象，而两个对象共享同一 Cache Line，会发生 False Sharing：

```text
Core 0 写 Slot A ─┐
                  ├─ 同一 Cache Line 在核心间来回失效
Core 1 写 Slot B ─┘
```

让每个槽位从 64B 边界开始，可避免**相邻小对象**共享缓存行。

但它不能解决：

- 两个核心真的同时访问同一个对象；
- 所有核心竞争全局 `head`；
- 跨 NUMA socket 访问远端内存。

### 7.2 全局 head 是扩展性瓶颈

算法是 lock-free，不代表没有竞争。每次 allocate/deallocate 都写同一个 `head` Cache Line：

```text
Core 0 ⇄ Core 1 ⇄ Core 2 ⇄ Core 3
              head cache-line bouncing
```

生产级方案：

```text
Per-Core Local Cache
    │ 本地 fast path：不碰全局 head
    │ 本地空/满时按 burst（如 32 个）交换
    ▼
Global Pool
```

这就是 DPDK `rte_mempool` 的关键思想：**per-lcore cache + bulk get/put + NUMA locality**。

## 八、复杂度与保证

| 项目 | 结果 |
|------|------|
| `allocate()` / `deallocate()` | 摊销 $O(1)$ |
| 热路径动态分配 | 0（但 `T` 构造函数自身仍可能分配） |
| 阻塞 | 算法无 mutex；CAS 竞争时重试 |
| Lock-free | 仅当目标平台相关 atomic 真正 lock-free |
| Wait-free | 否；某个线程可能长期输掉 CAS |
| ABA | 32-bit tag 降低风险，但 $2^{32}$ 次变更后会回绕 |

若要求数学意义上长期 ABA 安全，可考虑：

- 更宽 tag（可能需要 128-bit CAS）；
- Hazard Pointer；
- Epoch-Based Reclamation；
- 不复用节点，直到所有观察者离开；
- 按核心私有化，尽量消除共享 free list。

## 九、工业级边界条件

面试主动指出这些会加分：

1. **对齐：** 不能只把 `block_size` 向上取整；`raw_memory` 基地址也必须满足对齐。
2. **池生命周期：** 池销毁前必须归还所有对象。
3. **Double free：** 生产构建可用状态位、generation 或 debug bitmap 检测。
4. **Foreign pointer：** `deallocate()` 应验证地址范围、步长和槽位状态。
5. **异常：** Datapath 常禁异常并要求 `T` 构造/析构 `noexcept`；若允许构造异常，必须归还已 pop 的槽。
6. **析构：** 不允许析构函数抛异常。
7. **容量耗尽：** 返回 `nullptr` 后由上层选择 drop、backpressure 或统计告警，不能在关键核上阻塞。
8. **NUMA：** 内存在消费核心所在 node 分配并 first-touch；跨 socket 远程访问会显著增大延迟。
9. **大页：** 减少 TLB miss，但不自动解决 cache miss 和 head 竞争。
10. **基准：** 报 p50/p99/p999、不同核心数吞吐、CAS retry 数、LLC miss、跨 NUMA 情况，不能只报平均值。

## 十、面试追问速答

### Q1：它是 lock-free 还是 wait-free？

Lock-free：系统整体持续前进，但某线程可能一直 CAS 失败。不是 wait-free。

### Q2：为什么不直接用 mutex？

低竞争时 mutex 可能非常快；但 Datapath 追求可预测尾延迟，锁持有者被抢占会阻塞其他核心。是否采用无锁必须通过实际 contention benchmark 决定。

### Q3：tagged pointer 是否彻底解决 ABA？

有限位宽 tag 会回绕，因此是工程缓解，不是无限时间数学证明。并且 tag 本身不解决读取已结束生命周期节点的问题，所以实现还要保持 free-list 元数据永久有效，或使用 Hazard/Epoch。

### Q4：为什么每个槽位要永久 metadata？

避免 `Node` 与 `T` 复用同一字节区域造成对象生命周期和并发读取竞态。代价是每槽多几个字节，换取正确性与可调试性。

### Q5：为什么生产环境还要 Per-Core Cache？

无锁只消除了 mutex，不会消除全局原子变量上的 Cache Line Bouncing。Per-Core Cache 让绝大多数操作局部化，仅批量 refill/drain 时访问全局池。

### Q6：如何验证实现？

- 单线程：耗尽、复用、构造/析构次数；
- 多线程：高频 allocate/deallocate；
- 错误路径：double free、foreign pointer、构造异常；
- 工具：TSan、ASan/UBSan；
- 压测：不同核心数、CPU pinning、NUMA、本地池 vs 全局池；
- 观察：CAS retry、LLC miss、cache-to-cache transfer、p99/p999。

## 十一、白板答题顺序

1. 先写固定内存切块和 Placement New；
2. 写 free-list pop/push CAS；
3. 主动指出裸指针版本的 ABA；
4. 改成 `{tag,index}`；
5. 指出 metadata 不能被 `T` 覆盖；
6. 解释 acquire/release；
7. 最后扩展到 Per-Core Cache、bulk 操作、NUMA 和 HugePage。

一句话总结：

> 正确的 Datapath 内存池不仅是“atomic head + CAS”。还必须同时处理 ABA、节点生命周期、对齐、异常与误释放；规模扩大后，再用 per-core cache 和 bulk transfer 消除全局 head 的缓存行竞争。
