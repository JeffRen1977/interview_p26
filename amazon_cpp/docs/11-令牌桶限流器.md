# Part 21 — 惰性令牌桶限流器（EC2 数据面）

> 白板实现：[`examples/13_token_bucket_rate_limiter.cpp`](../examples/13_token_bucket_rate_limiter.cpp)

AWS VPC 里每个 EC2 实例挂一个高精度限流器，防止某个实例配置错误 / 被 DDoS / 恶意发包挤爆宿主机物理网卡。面试重点：

1. **不能开后台定时器线程**刷令牌 —— 百万 pps 下既不精准又拖垮流水线；
2. **惰性求值（Lazy Evaluation）**：只在收包瞬间按时间差补令牌；
3. **`double` 令牌**：防止微秒级亚令牌被整型截断；
4. **多核**：RSS 绑核（无锁）或 per-core local bucket。

---

## 一、令牌桶数学

| 参数 | 含义 |
|------|------|
| Capacity | 桶容量，决定允许的突发（burst）大小 |
| Rate | 稳态速率（pps 或 tokens/us） |

惰性补充：

$$
\text{refill} = (t_{\text{now}} - t_{\text{last}}) \times \text{rate}
$$

$$
\text{tokens} = \min(\text{capacity},\; \text{tokens} + \text{refill})
$$

每来一个包尝试扣 1 个令牌：够则放行，不够则丢弃。

---

## 二、为什么不用定时器线程

| 方案 | 问题 |
|------|------|
| 后台 timer 高频刷令牌 | 微秒级做不到精准；线程上下文切换开销大 |
| 每令牌一个定时事件 | 百万 pps → 百万事件，直接爆炸 |
| **惰性求值** | 零后台线程；开销只在收包路径，`O(1)` |

惰性的本质：**把“时间推进”折叠进“下一次收包”**。没有包到达时不需要算任何东西。

---

## 三、`double` vs 整型（高频追问）

1000 pps 换算到微秒：

$$
\text{rate} = \frac{1000}{1{,}000{,}000} = 0.001\ \text{token/us}
$$

若两包间隔仅几微秒：

- **整型**：`elapsed_us * rate` 恒为 0（截断）→ 令牌永不补充 → 高速下**误杀全部正常流量**；
- **`double`**：亚令牌增量可累加，攒满 1.0 再放行 → 限流曲线平滑。

> 面试金句：整型令牌桶在“高频小间隔”场景会退化成“永远空桶”。

## 四、实现要点与原始版本的修正

白板骨架：

```cpp
bool AllowPacket(uint64_t now_us) {
    if (now_us > last_update_us_) {
        double elapsed = now_us - last_update_us_;
        tokens_ = min(capacity_, tokens_ + elapsed * rate_per_us_);
    }
    last_update_us_ = now_us;          // 始终推进，避免令牌流失
    if (tokens_ >= 1.0) { tokens_ -= 1.0; return true; }
    return false;
}
```

相对常见示例修正了三点：

| 问题 | 说明 | 修正 |
|------|------|------|
| `last_update` 只在 `elapsed>0` 时更新 | 两包同一微秒时，`now == last`，虽无害；但若写成“仅 elapsed>0 才更新”，遇到未来还需累计的场景易漏更新 | 无条件推进 `last_update_us_` |
| 时钟回退 | `now < last`（NTP 跳变、跨核乱序）→ 无符号相减溢出成巨大值，瞬间灌满桶 | 仅 `now > last` 才补令牌，回退时不加 |
| 时间源硬编码 | 直接调 `steady_clock` 难测、且非真实数据面时钟 | `now_us` 作为参数注入（生产接 RDTSC / NIC 时间戳） |

关于时间源：

- 示范用 `steady_clock`（单调，适合测时延）；
- 极致场景映射 **RDTSC / NIC 硬件时间戳**；
- 用 RDTSC 要注意跨核一致性（`constant_tsc` / `nonstop_tsc`）与频率换算。

## 五、多核并发（彰显系统落地）

单实例多个 Data Core 并发收包时，这段代码存在 `tokens_` / `last_update_us_` 竞争。

| 方案 | 机制 | 代价 |
|------|------|------|
| 加锁 / atomic + CAS | 保护共享状态 | 高频竞争 → Cache line bouncing，拖垮吞吐 |
| **RSS 绑核** | 网卡按 flow hash 把同一实例流量导到同一核 | 数据面**天然单线程无锁**，性能最高 |
| **Per-Core Local Bucket** | 每核本地子桶先扣（无锁），耗尽再从全局桶批量申请（如 64） | 摊销后竞争概率降到约 1/N |

RSS 是首选；跨核不可避免时用 per-core + bulk refill（与无锁内存池的 Per-Core Cache 思路一致，见 [09](./09-无锁固定大小内存池.md)）。

## 六、扩展追问

### Q1：令牌桶 vs 漏桶（Leaky Bucket）？

| | 令牌桶 | 漏桶 |
|--|--------|------|
| 突发 | **允许**（攒令牌） | 平滑，不允许突发 |
| 输出 | 可突发到 capacity | 恒定速率流出 |
| 场景 | 网络限流常用（容忍 burst） | 需要严格整流时 |

### Q2：怎么支持字节级限流（bps）而不是 pps？

扣令牌时按包大小扣：`tokens_ -= packet_bytes`，rate 用 bytes/us。可再叠加一个 pps 桶做双维限流。

### Q3：capacity 如何选？

太小 → 正常突发被误杀；太大 → 失去限流意义、可能瞬时打爆下游。通常按“可容忍的最大突发 = rate × 允许突发时长”估。

### Q4：为什么不 sleep 到有令牌？

数据面**不能阻塞**：拒绝就立即丢包 / 反压，把等待决策交给上层（重传、拥塞控制）。

## 七、口述 45 秒版本

> 每个 EC2 实例挂令牌桶：rate 定稳态、capacity 定突发。绝不开定时器线程，而是惰性求值——收到包时用硬件时间戳差 × 速率补令牌，再 `min` 到 capacity。令牌用 `double`，否则 1000pps 换成 0.001 token/us，高频小间隔会被整型截断成永远空桶。时钟回退只推进时间、不加令牌。多核用 RSS 绑核做到无锁；必须跨核就用 per-core 本地桶 + 批量申请摊销竞争。
