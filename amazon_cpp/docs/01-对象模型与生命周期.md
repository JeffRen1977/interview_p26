## Part 1：C++ 对象模型（★★★★★）

### 对象生命周期

```cpp
void scope_demo() {
    Buffer b(1024);   // 构造：分配资源
    use(b);           // 使用
}                     // 析构：自动释放（栈展开）
```

| 阶段 | 发生什么 |
|------|----------|
| **默认构造** | 成员按声明顺序初始化 |
| **拷贝构造** | 用已有对象初始化新对象 `Buffer(const Buffer&)` |
| **移动构造** | 窃取资源 `Buffer(Buffer&&)` |
| **拷贝赋值** | `a = b` 已有对象 ← 已有对象 |
| **移动赋值** | `a = std::move(b)` 已有对象 ← 临时对象资源 |
| **析构** | 释放资源，逆序销毁成员 |

### Rule of Three / Five / Zero

| 规则 | 何时 | 需要定义 |
|------|------|----------|
| **Rule of Three** | C++11 前 | 析构、拷贝构造、拷贝赋值 |
| **Rule of Five** | C++11 后 | 上面 + 移动构造、移动赋值 |
| **Rule of Zero** | 推荐 | 成员全是 RAII 类型（`string`、`vector`、`unique_ptr`），编译器生成即可 |

```cpp
class Buffer {
    char* data_;
    size_t size_;
public:
    explicit Buffer(size_t n) : data_(new char[n]), size_(n) {}
    ~Buffer() { delete[] data_; }

    // 拷贝：深拷贝
    Buffer(const Buffer& o) : data_(new char[o.size_]), size_(o.size_) {
        std::copy(o.data_, o.data_ + size_, data_);
    }
    Buffer& operator=(const Buffer& o) {
        if (this != &o) {
            Buffer tmp(o);       // copy-and-swap：异常安全
            swap(tmp);
        }
        return *this;
    }

    // 移动：窃取指针
    Buffer(Buffer&& o) noexcept : data_(o.data_), size_(o.size_) {
        o.data_ = nullptr;
        o.size_ = 0;
    }
    Buffer& operator=(Buffer&& o) noexcept {
        if (this != &o) {
            delete[] data_;
            data_ = o.data_;
            size_ = o.size_;
            o.data_ = nullptr;
            o.size_ = 0;
        }
        return *this;
    }
};
```

### 高频问答

**Q: Copy Constructor 和 Assignment 的区别？**

| | 拷贝构造 | 拷贝赋值 |
|---|----------|----------|
| 时机 | **新对象**诞生 | **已有对象**被覆盖 |
| 语法 | `Buffer b2(b1);` | `b2 = b1;` |
| 旧资源 | 无（新对象） | 必须先释放 b2 旧资源 |

**Q: 为什么需要 Rule of Five？**

类管理堆资源时，编译器默认的拷贝是**浅拷贝** → 双析构 / double free。移动语义避免不必要的深拷贝。

**Q: Move Constructor 什么时候调用？**

- `Buffer b(std::move(x));`
- `return Buffer(n);` — 通常 RVO/NRVO，**不调用移动**（C++17 保证）
- 函数参数 `void f(Buffer b)` 传值时从临时移动
- `vector::push_back(Buffer{})` 若实现用移动

**Q: RVO / NRVO？**

```cpp
Buffer make() {
    return Buffer(1024);  // 直接在调用方栈帧构造，省略拷贝/移动
}
```

- **RVO**（Return Value Optimization）：返回无名临时量
- **NRVO**（Named RVO）：返回命名局部变量（编译器**可能**优化，C++17 后 mandatory copy elision 对 prvalue）
- 面试答：**优先依赖 RVO**；移动构造是 RVO 不可用时的后备

**Q: `explicit` / `=default` / `=delete`？**

```cpp
class Foo {
    explicit Foo(int);           // 禁止隐式 int → Foo
    Foo(const Foo&) = default;   // 显式使用编译器生成
    Foo& operator=(const Foo&) = delete;  // 禁止赋值（如单例）
};
```

## Part 4：引用（★★★★★）

### 值类别

| 类别 | 例子 | 绑定 |
|------|------|------|
| **lvalue** | 有名变量 `int x` | `T&` |
| **rvalue** | 临时量 `42`、`std::move(x)` | `T&&` |
| **xvalue** | 将亡值，如 `std::move(x)` 的结果 | `T&&` |

### 重载解析（必考）

```cpp
void foo(int&);        // ① lvalue
void foo(const int&);  // ② const lvalue / rvalue（万能引用前的后备）
void foo(int&&);       // ③ rvalue

int x = 1;
foo(x);              // ① int&
foo(1);              // ③ int&&
foo(std::move(x));   // ③ int&&
const int cx = 2;
foo(cx);             // ② const int&（cx 是 const lvalue，不能绑 int&）
```

**规则：** 非 const lvalue 优先绑 `T&`；rvalue 优先绑 `T&&`；const lvalue 绑 `const T&`。

## Part 5：Move Semantics（★★★★★）

### std::move 做了什么？

```cpp
std::move(x)  // 等价于 static_cast<T&&>(x)
```

**只做类型转换，不移动任何东西。** 告诉编译器「可以窃取 x 的资源」，实际移动发生在移动构造/赋值被调用时。

### 为什么 std::move 不移动？

移动是**重载解析 + 构造函数/赋值运算符**的行为；`std::move` 只是 cast。

### Perfect Forwarding

```cpp
template <typename T>
void wrapper(T&& arg) {
    work(std::forward<T>(arg));  // 保持 arg 原有的值类别
}
```

| 调用 | T 推导 | forward 转给 work |
|------|--------|-------------------|
| `wrapper(x)` | `int&` | `int&` |
| `wrapper(1)` | `int` | `int&&` |

`T&&` + 引用折叠 + `std::forward` = 完美转发。用于 `make_unique<T>(args...)`、`emplace_back`。

## Part 6：RAII（★★★★★）

**Resource Acquisition Is Initialization** — 在构造函数获取资源，在析构函数释放。

```cpp
{
    std::lock_guard<std::mutex> lock(mtx);  // 构造时 lock
    shared_data++;
}   // 析构时 unlock，即使抛异常也释放
```

### 高频问答

**Q: 为什么 Exception Safe？**

栈展开（stack unwinding）时自动调用已构造对象的析构函数 → 锁、文件、内存必定释放。

**Q: lock_guard 为什么不会死锁？**

- 单锁：`lock_guard` 析构必 unlock，无手动 `unlock` 遗漏
- 多锁：应用 `std::lock(m1, m2)` + `lock_guard` adopt_lock，固定加锁顺序

**Q: unique_ptr 为什么属于 RAII？**

构造时拥有堆对象，析构时 `delete`；不可拷贝，所有权唯一。

## Part 7：智能指针（★★★★★）

### 三者对比

| | unique_ptr | shared_ptr | weak_ptr |
|---|------------|------------|----------|
| 所有权 | 独占 | 共享（引用计数） | 不拥有，观察 |
| 拷贝 | ❌ | ✅ 计数+1 | ✅ 不增加 strong count |
| 大小 | 8 bytes | 16 bytes（ptr + control block） | 16 bytes |
| 场景 | 默认首选 | 共享所有权 | 打破循环引用、缓存 |

### shared_ptr 底层

```
shared_ptr object          Control Block (heap)
┌─────────────┐           ┌──────────────────┐
│ raw ptr ────┼──────────→│ strong count     │
│ ctrl blk ptr┼──────────→│ weak count       │
└─────────────┘           │ deleter          │
                          │ allocator        │
                          └──────────────────┘
```

- `make_shared<T>()`：一次分配 object + control block，更 cache-friendly
- 引用计数的 inc/dec 是 **atomic**（线程安全）；`*ptr` 读写需用户同步

### weak_ptr 为什么存在？

```cpp
struct A { std::shared_ptr<B> b; };
struct B { std::shared_ptr<A> a; };  // 循环引用 → 泄漏
// 修复：一方改 std::weak_ptr<A>
```

### enable_shared_from_this

在对象已被 `shared_ptr` 管理时，安全地返回 `shared_ptr<this>`：

```cpp
class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getPtr() { return shared_from_this(); }
};
// 必须先有 shared_ptr 拥有 Widget，才能调用 shared_from_this()
```
