## Q1：Implement `std::shared_ptr` from Scratch

**题目：** 手写简化版 `shared_ptr`，考察引用计数、control block、线程安全。

### 1.1 核心概念

| 概念 | 说明 |
|------|------|
| **Managed object** | 用户看到的 `T*` 所指对象 |
| **Control block** | 堆上结构：strong count、weak count（可选）、deleter、allocator |
| **为什么分开** | 多个 `shared_ptr` 可共享 control block；`enable_shared_from_this` 需要独立 block |
| **线程安全** | **引用计数的 inc/dec 必须原子**；所指对象的读写不自动线程安全 |

### 1.2 简化实现（面试手写版）

```cpp
#include <atomic>
#include <cstddef>
#include <utility>

template <typename T>
class SharedPtr {
 public:
  SharedPtr() : ptr_(nullptr), ctrl_(nullptr) {}

  explicit SharedPtr(T* p) : ptr_(p) {
    if (p) ctrl_ = new ControlBlock{p};
  }

  ~SharedPtr() { release(); }

  SharedPtr(const SharedPtr& other) : ptr_(other.ptr_), ctrl_(other.ctrl_) {
    inc();
  }

  SharedPtr& operator=(const SharedPtr& other) {
    if (this != &other) {
      release();
      ptr_ = other.ptr_;
      ctrl_ = other.ctrl_;
      inc();
    }
    return *this;
  }

  SharedPtr(SharedPtr&& other) noexcept
      : ptr_(other.ptr_), ctrl_(other.ctrl_) {
    other.ptr_ = nullptr;
    other.ctrl_ = nullptr;
  }

  T& operator*() const { return *ptr_; }
  T* operator->() const { return ptr_; }
  T* get() const { return ptr_; }
  long use_count() const { return ctrl_ ? ctrl_->strong.load() : 0; }

 private:
  struct ControlBlock {
    std::atomic<long> strong{1};
  };

  T* ptr_;
  ControlBlock* ctrl_;

  void inc() {
    if (ctrl_) ctrl_->strong.fetch_add(1, std::memory_order_relaxed);
  }

  void release() {
    if (!ctrl_) return;
    if (ctrl_->strong.fetch_sub(1, std::memory_order_acq_rel) == 1) {
      delete ptr_;
      delete ctrl_;
    }
  }
};
```

### 1.3 面试必须讲出的点

| 问题 | 答案 |
|------|------|
| `shared_ptr` 大小 | 通常 **16 bytes**（64-bit）：raw ptr + control block ptr |
| 线程安全吗？ | **引用计数原子操作线程安全**；`*ptr` 读写需用户同步 |
| `memory_order` | `fetch_sub` 用 **acq_rel**：dec 到 0 时 synchronize-with 其他线程的 release |
| 循环引用 | `A↔B` shared_ptr 泄漏 → 用 `weak_ptr` 打破 |
| `make_shared` | 一次分配 object + control block 连续内存，更 cache-friendly |
| 性能 | 原子 inc/dec 有开销；热路径考虑 `unique_ptr` 或 raw + 明确 owner |

### 1.4 追问：`weak_ptr` 做什么？

- 不增加 strong count  
- `lock()` 尝试 promote 为 `shared_ptr`（原子读 strong > 0）  
- 用于缓存、观察者、打破 cycles

## Q2：Virtual Functions and vTables

**题目：** 运行时多态在内存里怎么工作？vptr、vtable、性能开销？

### 2.1 机制

```cpp
class Base {
 public:
  virtual void foo();
  virtual ~Base() = default;
  int x;
};

class Derived : public Base {
 public:
  void foo() override;
  int y;
};
```

**内存布局（典型 Itanium C++ ABI）：**

```
Derived object:
┌──────────────┐
│ vptr ────────┼──→ Derived vtable
├──────────────┤      ┌─────────────┐
│ Base::x      │      │ &Derived::foo │
├──────────────┤      │ &Derived::~D  │
│ Derived::y   │      │ ...typeinfo   │
└──────────────┘      └─────────────┘
```

| 概念 | 说明 |
|------|------|
| **vptr** | 编译器在 **有虚函数的对象开头** 插入 hidden pointer |
| **vtable** | 每个 **有虚函数的类** 一张静态表，存函数指针 |
| **调用过程** | `obj->foo()` → 读 vptr → 查 vtable[index] → 间接跳转 |
| **构造顺序** | 构造 Base 时 vptr 先指向 Base vtable，Derived  ctor 完成后才指向 Derived vtable |

### 2.2 性能开销

| 开销 | 量级 |
|------|------|
| 额外内存 | 每对象 +1 vptr（8 bytes on 64-bit） |
| 调用 | **2 次间接寻址** + 间接分支 → 可能 **ICache miss / branch mispredict** |
| 内联 | 虚调用 **不可 devirtualize**（除非 LTO / final / `-devirtualize`） |
| 对比 | 非虚调用：直接 call 或内联 |

**AV 场景：** 传感器 driver 接口常用虚函数做 plugin；**热路径**（每帧百万次）应用 CRTP 或 function pointer + 单 impl 避免 vcall。

### 2.3 常见追问

| 问题 | 答案 |
|------|------|
| 纯虚函数 | vtable 条目为 nullptr（或 __cxa_pure_virtual） |
| 多继承 | 可能有 **多个 vptr**；cross-cast 需调整 this pointer |
| `virtual` destructor | 基类指针 delete derived 时必须 virtual，否则 UB |
| RTTI | `type_info` 常与 vtable 关联；`dynamic_cast` 更贵 |

## Q3：`volatile`, `const`, and `inline`

### 3.1 `volatile`

**语义：** 每次读写必须 **落到内存**，编译器 **不得** 优化掉、不得缓存到寄存器。

**正确使用场景：**

| 场景 | 示例 |
|------|------|
| **Memory-mapped I/O** | `*(volatile uint32_t*)0x40001000 = 1;` 写硬件寄存器 |
| **Signal handler flag** | `volatile sig_atomic_t stop = 0;`（简单 flag） |
| **Dual-port RAM / HW FIFO** | 状态寄存器轮询 |

**错误使用：**

| 误用 | 为什么错 |
|------|----------|
| 多线程同步 | `volatile` **不提供** atomicity 或 memory ordering → 用 `std::atomic` |
| 「防止优化」普通变量 | 应使用 atomic 或 compiler barrier |

**面试答：** 在 DRIVE 嵌入式里，`volatile` 用于 **MMIO 寄存器** 和 **ISR 与主循环间的简单 flag**；线程同步一律 `mutex` / `atomic`。

### 3.2 `const`

| 用法 | 含义 |
|------|------|
| `const int x` | 不可修改 |
| `const int* p` | 所指不可改；指针可改 |
| `int* const p` | 指针不可改；所指可改 |
| `const int* const p` | 都不可改 |
| 成员函数 `void foo() const` | `this` 为 `const T*`，不能改非 mutable 成员 |

**嵌入式：** `const` 数据放 flash（ROM），减 RAM；驱动 lookup table 常用 `const`。

### 3.3 `inline`

| 层面 | 说明 |
|------|------|
| C++ `inline` 关键字 | **ODR 豁免**（头文件定义 multiple TU）+ **建议**内联 |
| 编译器决定 | 实际是否 inline 看 `-O2`、函数大小、虚函数不行 |
| 热路径 | 小函数 `getTimestamp()` 可 inline 消除 call overhead |

## Q4：Mutex vs Spinlock vs Semaphore

### 4.1 对比表

| 原语 | 行为 | 适用场景 |
|------|------|----------|
| **Mutex** | 拿不到就 **sleep**（OS 调度） | 临界区 **较长**、竞争不极端 |
| **Spinlock** | 拿不到就 **busy-wait** 自旋 | 临界区 **极短**、**低延迟 RT**、多核 |
| **Semaphore** | 计数器；P/V 操作；可 >1 | **资源池**（N 个 buffer slot） |
| **Condition Variable** | 配合 mutex 等待条件 | 生产者-消费者 |

### 4.2 何时用 Spinlock 而非 Mutex？

**答（RT 内核线程典型场景）：**

1. 临界区 **只有几条指令**（几个 pointer 赋值）  
2. 不能 tolerate **调度延迟**（mutex sleep 可能 μs–ms 级）  
3. 运行在 **已知多核** 上，且持有时间 **远小于 time slice**  
4. 已在 **中断上下文** 或 **real-time priority** 线程（mutex sleep 可能优先级反转）

**风险：**  
- 单核上 spinlock + 持有者被抢占 → **死锁**（需 disable preemption 或 adaptive spin）  
- 长时间自旋浪费 CPU、功耗、影响同核其他任务  

**AV 例子：** Sensor ring buffer 的 head/tail 更新（几条 atomic 指令）→ `std::atomic` 或短 spinlock；大块 JSON 解析 → mutex。

### 4.3 `std::atomic` vs Spinlock

很多「短临界区」直接用 **lock-free atomic**（`fetch_add`, CAS）比 spinlock 更好。

```cpp
// 无锁 ring buffer 索引更新示意
std::atomic<size_t> head{0}, tail{0};
```

### 4.4 其他常考同步

| 概念 | 要点 |
|------|------|
| **Priority inversion** | 低优先级持锁，高优先级等 → **priority inheritance mutex** |
| **Reader-writer lock** | 读多写少（config 表） |
| **Memory order** | relaxed/acquire/release/seq_cst 选用 |

## Q5：Bonus — 嵌入式常追问

| 话题 | 要点 |
|------|------|
| **Cache coherency** | DMA buffer 需 **cache flush/invalidate** 或 **uncached mapping** |
| **Memory alignment** | SIMD / DMA 要求 16/32/64-byte align |
| **Stack vs heap RT** | RT path 避免动态分配；对象池 |
| **False sharing** | 两 atomic 同一 cache line → padding |
| **Linux vs QNX** | AV 常用 QNX 做 safety-critical partition |
