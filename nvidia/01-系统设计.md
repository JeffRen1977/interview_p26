# 01 - 高性能系统设计（NVIDIA DRIVE）

> **面试风格：** 不是 Design Twitter，而是 **高吞吐、低延迟、硬件感知** 的嵌入式/车载/数据中心系统。

---

## Q1：Design a Log Ingestion Pipeline for Test Vehicles

**题目：** AV 测试车队每小时产生 TB 级 camera / radar / LiDAR 数据。如何在车辆 **dock 或有 LTE/5G** 时，高效地 stream、compress、serialize、store？

### 1.1 需求澄清（先问面试官）

| 维度 | 典型假设 |
|------|----------|
| 数据量 | 6–12 cameras @ 30fps 4K RAW/YUV；2 LiDAR @ 10–20Hz；3 radar；IMU/GPS/CAN @ kHz |
| 峰值带宽 | 车内 NVMe 写入 2–4 GB/s；LTE ~50 Mbps；5G ~500 Mbps；Wi-Fi 6 dock ~1 Gbps |
| 约束 | 录制时不能丢帧；upload 可后台；crash-safe；可审计（legal/compliance） |
| 用户 | 工程师 replay debug；ML 训练 pipeline 消费 |

### 1.2 车内录制架构（Hot Path — 行驶中）

```
Sensors → Drivers → Timestamp + Sequence ID (PTP/gPTP sync)
              ↓
        Ring Buffer Pool (zero-copy, DMA/GPU memory)
              ↓
        Recorder Service (async writer threads)
              ↓
        Local NVMe (chunked files + index)
```

**关键设计点：**

| 组件 | 设计 |
|------|------|
| **时间戳** | 每个 sample 带 `sensor_id, hw_timestamp_ns, seq_num, rig_frame_id` |
| **零拷贝** | Camera/LiDAR buffer 用 ref-counted blocks；多 consumer（recorder + live perception）共享 |
| **序列化** | 优先 **容器格式**（MCAP / custom chunk）而非 per-frame protobuf；metadata sidecar |
| **压缩** | Camera：H.265 HW encoder 或 JPEG-XL；LiDAR：量化 xyz + LZ4/ZSTD；Radar：原始或轻量压缩 |
| **写入** | 双缓冲 + 独立 IO 线程；顺序写大 chunk（64–256 MB）减少 fs 开销 |
| **完整性** | Per-chunk CRC32；seq gap detection；crash 时最后 chunk 可 truncated 但 index 可修复 |
| **Backpressure** | Recorder 跟不上 → 降非关键流码率 / 丢 preview 而非 safety-critical raw |

**文件布局示例：**

```
/session_20250623_143022/
  manifest.json          # rig calib, sensor list, software version
  index.db               # (timestamp → file, offset, size, sensor_id)
  cam_front_raw_0000.mcap
  lidar_top_0000.mcap
  can_bus_0000.log
  checksums.sha256
```

### 1.3 Upload Pipeline（Dock / 5G）

```
Local Index Scanner → Priority Queue (新 session / 标注需求 / 失败重传)
        ↓
Upload Agent (multi-thread, resume-capable)
        ↓
TLS + multipart upload → Object Storage (S3/GCS compatible)
        ↓
Ingestion Service → Catalog (DB) → Trigger ML training / sim replay
```

| 策略 | 说明 |
|------|------|
| **Delta / 增量** | 只上传未 sync 的 chunk |
| **Resume** | 断点续传；upload_id + byte offset |
| **优先级** | Safety event clip > 完整 session |
| **Edge 预处理** | Dock 时可 GPU transcode（4K RAW → 训练友好格式）再上传，减带宽 |
| **加密** | 静态 AES-256；传输 TLS 1.3 |

### 1.4 延迟与吞吐估算（面试加分）

示例：8 × 4K YUV422 @ 30fps ≈ 8 × 3840×2160×2 × 30 ≈ **3.8 GB/s** raw  
→ 必须 HW encode（H.265 ~50:1）→ ~**80 MB/s** 才可行于 NVMe 长期录制。

**口述模板：**

> 车内录制以 **不丢帧 + crash-safe + 时间对齐** 为第一目标；upload 是 best-effort 后台任务。我用 chunk + index 分离、零拷贝 buffer pool、HW 压缩，dock 后按优先级增量同步到 object storage。

## Q2：Design a High-Performance GPU Job Scheduler

**题目：** 如何在 **数千 GPU** 上编排 DL 训练任务，最大化吞吐、减少 idle？

### 2.1 需求澄清

| 维度 | 假设 |
|------|------|
| 工作负载 | 分布式训练 job（多 node × 8 GPU）；hyperparameter sweep；推理 batch job |
| 目标 | GPU utilization > 85%；job 完成时间（JCT）短；公平性 / 优先级 |
| 约束 | 网络（NCCL）、本地 NVMe、GPU 拓扑（NVLink）、故障恢复 |

### 2.2 架构

```
Submit API (REST/gRPC)
    ↓
Scheduler Control Plane
  ├─ Job Queue (priority, gang scheduling)
  ├─ Resource Manager (GPU/CPU/mem/network inventory)
  ├─ Placement Engine (bin packing + topology-aware)
  └─ Preemption / Checkpoint policy
    ↓
Agent on each node (Daemon)
  ├─ GPU health / metrics
  ├─ Container / MIG / K8s pod launcher
  └─ Local cache (dataset staging)
    ↓
Training Workers (PyTorch + NCCL)
```

### 2.3 核心算法与策略

| 策略 | 说明 |
|------|------|
| **Gang scheduling** | 分布式 job 需同时获得 N 个 GPU，避免 partial allocation 死等 |
| **Topology-aware placement** | 同一 job 优先同一 node 内 NVLink 互联 GPU |
| **Bin packing** | First-Fit Decreasing / Tetris：按 GPU mem 需求填坑 |
| **Gang + backfill** | 大 job 等待时，用小 job 填空闲 GPU（不破坏 gang 约束时用 backfill） |
| **Preemption** | 低优先级 job checkpoint → 让路给高优先级 |
| **Data locality** | 数据集已在 node 本地 NVMe 时优先调度到该 node |
| **Fault tolerance** | GPU fail → 自动 replace rank；elastic training |

### 2.4 避免 GPU Idle 的具体手段

1. **Right-sizing：** 小模型不用整 node 8 卡，可 MIG 或多 job 共享  
2. **Pipeline overlap：** Dataloader prefetch、gradient all-reduce overlap（软件层，但 scheduler 要留 CPU/NIC 余量）  
3. **Sweep 批处理：** 合并小实验为单个 multi-trial job  
4. **Queue 预测：** 根据历史 job 时长预测，提前 warm pool  
5. **Metrics 闭环：** Prometheus 监控 `gpu_util`, `sm_active`, `nccl_wait` → 调度策略 A/B  

### 2.5 面试答题结构

```
1. Control plane vs data plane 分离
2. Job 模型：resource vector (gpu_count, gpu_mem, nic, duration_hint)
3. Placement：gang + topology + locality
4. Preemption + checkpoint 减少 head-of-line blocking
5. Observability：利用率、JCT、排队时间 SLI
6. Failure：node drain、自动重试
```

**与 DRIVE 岗联系：** 即使不做 cloud scheduler，也体现 **并行资源编排、避免 bubble、硬件拓扑感知** — 和 sensor pipeline 多 consumer 调度同源。

## Q3：Real-Time Sensor Fusion Architecture

**题目：** 8 cameras + 3 radars + 2 LiDARs，时间同步 + 融合，**端到端 < 50–100 ms** 给 perception。

### 3.1 系统架构

```
┌─────────────────────────────────────────────────────────┐
│  Sensors (PTP/gPTP time sync, hardware trigger optional) │
└────────┬──────────┬──────────┬──────────────────────────┘
         │          │          │
    Camera ISP   Radar drv   LiDAR drv
         │          │          │
         └──────────┴──────────┴──→ Time Sync Service
                        │
              Sensor Data Service (DriveWorks SAL-like)
              - unified timestamp domain
              - sequence / frame_id
              - ref-counted buffer publish
                        │
         ┌──────────────┼──────────────┐
         ▼              ▼              ▼
   Camera preproc   Radar DSP    LiDAR motion comp
   (CUDA: debayer,  (CFAR,        (deskew using
    undistort)       clustering)    ego motion)
         │              │              │
         └──────────────┴──────────────┘
                        │
              Fusion / Sync Buffer
              (align to t_ref via interpolation)
                        │
              Perception (BEV / multi-modal net)
                        │
              Output: unified world model @ t_ref
```

### 3.2 时间同步

| 机制 | 说明 |
|------|------|
| **PTP (IEEE 802.1AS)** | 全网时钟对齐到 μs 级 |
| **Per-sensor timestamp** | 采集时刻 hw timestamp，非 receive 时刻 |
| **Frame ID** | 逻辑帧号，便于 cross-sensor 关联 |
| **Reference time $t_{ref}$** | 通常选主 camera mid-exposure 或 LiDAR spin center |
| **对齐策略** | 最近邻 / 线性插值（IMU）/ motion compensation（LiDAR 点云去畸变） |

**LiDAR motion distortion：** 一帧扫描 50–100 ms 内 ego 在运动 → 用 IMU/odometry 把点云 warp 到统一 $t_{ref}$。

### 3.3 延迟预算（目标 80 ms 示例）

| 阶段 | 预算 | 说明 |
|------|------|------|
| Sensor exposure → DMA complete | 5–15 ms | rolling shutter 更长 |
| Driver + publish | 2–5 ms | zero-copy |
| Preprocessing (GPU) | 5–10 ms | 8 cam 可 batch |
| Time sync + buffer wait | 5–10 ms | 等待最慢 sensor |
| Fusion align | 2–5 ms | |
| Perception inference | 25–40 ms | TensorRT on Orin DLA/GPU |
| Post-process + output | 5–10 ms | |
| **Total** | **~50–95 ms** | 留 jitter margin |

### 3.4 关键工程决策

| 问题 | 推荐 |
|------|------|
| 同步 vs 异步 | **Sync frame** 给 perception；async stream 给 logging |
| 8 camera 并行 | GPU batch inference；NVIDIA SIPL / zero-copy NVSciBuf |
| 某 sensor 延迟 | timeout → 用上一帧 / 降权 / safe mode |
| CPU vs GPU | 预处理上 GPU；fusion 逻辑可 CPU，heavy math 上 CUDA |
| 确定性 | RTOS partition (QNX) + pinned thread priority |

### 3.5 面试金句

> Fusion 的核心不是简单「同一 timestamp 拼在一起」，而是 **统一时间域 + motion compensation + 延迟预算可证明**。Perception 消费的是 $t_{ref}$ 时刻的世界快照，各 sensor 数据都 warp 到这个时刻。

## 系统设计答题模板（通用）

```
1. 澄清：规模、SLA、硬件、failure mode
2. 架构图：data plane vs control plane
3. 热路径：零拷贝、批处理、HW 加速
4. 数字：带宽/延迟预算表
5. 可靠性：backpressure、fallback、crash recovery
6. 观测：metrics、logging、debug replay
7. Trade-off：一致性 vs 延迟 vs 成本
```
