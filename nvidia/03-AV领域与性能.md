## Q1：Behavior and Motion Planning — Hybrid A*

**题目：** 遇到 **double-parked delivery truck** 挡路，A* / Hybrid A* 如何算替代轨迹？

### 1.1 问题分解

| 层级 | 任务 |
|------|------|
| **Behavior** | 决策：等待 / 换道 / 借道绕行 / 重规划 |
| **Motion Planning** | 在决策后生成 **可行、舒适、无碰撞** 的 path + speed profile |
| **Obstacle** | 静态阻塞（double-park）、动态（oncoming traffic） |

Double-park 场景典型决策：**借对向车道绕行**（gap 足够）或 **减速等待**（gap 不足）。

### 1.2 经典 A*（几何路径，离散 grid）

**状态：** grid cell $(x, y)$  
**代价：** $f(n) = g(n) + h(n)$

- $g(n)$：从起点到当前的实际代价（距离、曲率惩罚）  
- $h(n)$：启发式到目标（Euclidean / Manhattan），需 **admissible**（不高估）

**局限：** 车辆是 **非完整约束**（Non-holonomic）——不能原地横移，纯 grid A* 会产生「锯齿」不可行路径。

### 1.3 Hybrid A*（停车场 / 低速 / 结构化道路常用）

**核心思想：** 在 **连续状态空间** $(x, y, \theta)$ 离散化，但用 **车辆运动学模型** 扩展节点。

**状态：** $(x, y, \theta)$  
**扩展：** 用 Reeds-Shepp 或 Dubins 曲线连接，满足最小转弯半径 $R_{min}$

**运动学（自行车模型简化）：**

$$
\dot{x} = v \cos\theta,\quad \dot{y} = v \sin\theta,\quad \dot{\theta} = \frac{v}{L}\tan\delta
$$

**算法流程：**

```
1. 从 ego 当前 (x,y,θ) 作为 start node
2. Priority queue by f = g + h
3. Pop node → 用 motion primitives 扩展 (v, δ) 离散集合
4. 碰撞检测：occupancy grid / convex hull vs truck polygon
5. 到达 goal region → 可选 analytic expansion (Reeds-Shepp 直连)
6. 回溯得 path
7. 后处理：平滑 (splines) + speed planning (ST graph / jerk limits)
```

**Double-park 绕行：**

```
1. Perception 提供 truck occupancy + lane graph
2. Behavior：选择 left bypass（检查对向车道 gap）
3. Hybrid A* 在 (x,y,θ) 搜索，代价含：
   - 偏离参考线惩罚
   - 接近障碍物惩罚
   - 换道 / 方向盘角速度惩罚（comfort）
4. 输出 path → ST graph 上对动态 obstacle 做 speed profile
5. Controller 跟踪 trajectory
```

### 1.4 面试口述模板

> 纯 A* 在 $(x,y)$ grid 上不考虑车辆曲率约束。Hybrid A* 在 $(x,y,\theta)$ 上用运动学原语扩展，用启发式 Reeds-Shepp 距离。Double-park 时 behavior 层先决定借道，motion 层在 occupancy map 上搜索无碰撞 path，再加 comfort 代价和平滑。

### 1.5 相关算法（加分）

| 算法 | 场景 |
|------|------|
| **RRT\*** | 复杂 unstructured 空间 |
| **Lattice planner** | 高速公路结构化 |
| **EM planner (Apollo)** | 分 path + speed 两阶段 |
| **Optimization (iLQR / MPC)** | 精细轨迹、动态约束 |

## Q2：Quantize FP32 → INT8 for DRIVE Orin/Thor

**题目：** 如何把网络量化到 INT8，在 **DRIVE Orin/Thor** 上跑满 FPS？

### 2.1 Orin/Thor 推理栈（NVIDIA）

```
PyTorch / ONNX model
    ↓
TensorRT (或 cuDNN + DLA compiler)
    ↓  INT8 calibration / QAT
Engine (GPU CUDA cores + DLA optional)
    ↓
Drive OS / CUDA runtime on Orin
```

| 硬件 | 特点 |
|------|------|
| **GPU (Ampere/Ada)** | 通用 INT8/FP16 Tensor Core |
| **DLA** | Dedicated deep learning accelerator，低功耗固定 op |
| **Orin** | 254 TOPS (INT8 sparse) 级（视 SKU） |
| **Thor** | 下一代，更高集成度 |

### 2.2 量化流程

```
1. FP32 baseline → 精度 / latency benchmark
2. 选择量化策略：
   - PTQ：校准集 500–1000 帧（覆盖场景）
   - QAT：精度敏感模型
3. TensorRT build：
   - INT8 calibrator (entropy / minmax)
   - sensitive layers → FP16 fallback
4. 部署：
   - GPU engine 或 DLA（部分 subgraph）
5. 验证：
   - mAP / IoU / 感知指标 + 实车 corner case
```

**量化公式：**

$$
q = \mathrm{clip}\left(\mathrm{round}\left(\frac{x}{s}\right) + z\right)
$$

详见 [docs/07-端侧部署题详解.md](../docs/07-端侧部署题详解.md)。

### 2.3 Orin 特有 trade-off

| 决策 | 说明 |
|------|------|
| **GPU vs DLA** | DLA 省电但 op 支持有限；不支持 op fallback GPU |
| **INT8 vs FP16** | 检测头 / BEV transformer attention 可能需 FP16 |
| **Dynamic shape** | 尽量 fixed shape batch=1 深度优化 |
| **Multi-stream** | 8 camera batch inference 提高 GPU 利用率 |
| **Pipeline** | 感知 | 规划并行不同 engine |

### 2.4 敏感层处理

| 层 | 策略 |
|----|------|
| 第一层 Conv | 常保留 FP16 |
| BEV pooling / scatter | 精度敏感 |
| Softmax attention | FP16 |
| Detection head | per-channel quant 或 FP16 |

**口述模板：**

> 我先 TensorRT PTQ 出 INT8 baseline，用感知 KPI 回归集验证。掉点层保留 FP16 或做 QAT。Orin 上把 backbone batch 8 cam 送 GPU，head 走 DLA 省功耗——以 profiling 数据定，不用猜。

## Q3：Amdahl's Law — GPU 并行加速上限

**题目：** 把一段代码并行到 GPU，**整个 pipeline** 最多能快多少？

### 3.1 公式

$$
S_{max} = \frac{1}{(1 - P) + \frac{P}{N}}
$$

| 符号 | 含义 |
|------|------|
| $P$ | 可并行部分占整体时间的比例（0–1） |
| $N$ | 并行处理器数量（GPU SM / 整卡倍数） |
| $1-P$ | **串行瓶颈**（Amdahl 瓶颈） |

**极限 $N \to \infty$：**

$$
S_{max} \to \frac{1}{1 - P}
$$

### 3.2 手算例题

**例 1：** 程序 80% 可 GPU 并行，20% 必须 CPU 串行，无限 GPU：

$$
S_{max} = \frac{1}{0.2 + 0} = 5\times
$$

**例 2：** 感知 pipeline：预处理 30ms (GPU) + 推理 50ms (GPU) + 后处理 10ms (CPU) + IPC 10ms (串行)

总 100ms，可并行 80ms → $P = 0.8$

$$
S_{max} = \frac{1}{0.2 + 0.8/N}
$$

$N=4$：$S = 1/(0.2 + 0.2) = 2.5\times$ → 100ms → **40ms**  
即使 GPU 无限快：$S = 5\times$ → **20ms**（10ms 串行无法消除）

### 3.3 面试要点

| 要点 | 说明 |
|------|------|
| **串行部分** | PCIe copy、sync、单线程后处理、logging |
| ** Gustafson's Law** | 若问题规模随核数增长，加速比可以更好——但 AV 实时 pipeline 常是 **固定 deadline** |
| **工程行动** | 优化 $1-P$：async copy、pipeline stage、DLA+GPU overlap |
| **Profiling first** | 用 Nsight 找 true bottleneck，避免优化已很小的部分 |

**金句：**

> GPU 再快，串行 20% 就把加速上限钉死在 5×。我会先 profile 找 $1-P$，优化 PCIe、CPU 后处理和 frame sync，再考虑 kernel 微优化。
